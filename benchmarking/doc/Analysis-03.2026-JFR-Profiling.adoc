[[_why_cpu_doesnt_match_latency]]
== Why CPU% Doesn't Match Latency%

JFR shows CUI library at 0.3% of CPU samples, yet the ablation study measures JWT validation at 46% of request time. This is explained by **CPU saturation**: with 114 avg threads competing for 10 cores (~11:1 oversubscription at 85% peak CPU), most threads are **off-CPU** (waiting to be scheduled) at any given JFR sample point. JFR `ExecutionSample` only captures on-CPU time, but wall-clock latency includes both on-CPU execution and CPU scheduling wait time. A JMH concurrency sweep confirms: RSA latency scales proportionally to thread oversubscription (48µs at 1 thread → 556µs at 50 threads on HotSpot JVM). See link:Analysis-03.2026-Integration.adoc#_root_cause_cpu_saturation_from_rsa_under_concurrency[root cause analysis].

== Access Log Filter

The `CustomAccessLogFilter` runs on all JAX-RS endpoints. Measured impact (filter disabled):

* Health P50: 0.510ms → 0.417ms (-0.093ms)
* JWT P50: 1.91ms → 1.96ms (within variance)

The filter does not explain the JWT-health gap. It can be disabled for benchmark isolation: `cui.http.access-log.filter.enabled=false`.

[[_cdi_producer_chain_research_findings]]
== CDI Producer Chain Research

This section documents why the CDI `Instance.get()` overhead cannot be reduced.

=== What `Instance.get()` Does

Each call (from Quarkus ArC source):

1. Creates child `CreationalContext`
2. Constructs `InjectionPointImpl` with type, qualifiers, annotations
3. Sets `InjectionPoint` via `InjectionPointProvider.setCurrent()`
4. Creates bean instance (`@Dependent` scope)
5. Cleans up in try-finally

JFR confirms: 250 `CreationalContextImpl` allocations during benchmark. The ablation study measures the total cost at ~20µs — fast individually.

=== Why `@WithCaching` Cannot Be Applied

Quarkus ArC's `@WithCaching` caches `Instance.get()` results. The `JwtValidationEndpoint` uses 5 fields with different `@BearerToken` parameters:

[source,java]
----
@BearerToken Instance<BearerTokenResult> basicToken;
@BearerToken(requiredScopes = {"read"}) Instance<BearerTokenResult> tokenWithScopes;
@BearerToken(requiredRoles = {"user"}) Instance<BearerTokenResult> tokenWithRoles;
@BearerToken(requiredGroups = {"test-group"}) Instance<BearerTokenResult> tokenWithGroups;
@BearerToken(requiredScopes = {"read"}, requiredRoles = {"user"}, ...) Instance<BearerTokenResult> tokenWithAll;
----

Caching would return `basicToken`'s result for all fields — a security bypass. The `@Dependent` scope with per-request `InjectionPoint` introspection is required for correct authorization.

=== Memory Leak Risk

`Instance.get()` for `@Dependent` beans in `@ApplicationScoped` holders accumulates dependent instances for the application lifetime. Mitigate with `Instance.Handle` or `instance.destroy()`.

== Container Configuration

JFR output requires a volume mount:

[source,yaml]
----
# docker-compose.jfr.yml
services:
  token-sheriff-integration-tests:
    image: "token-sheriff-integration-tests:jfr"
    volumes:
      - ./target/jfr-output:/tmp/jfr-output:rw
----
