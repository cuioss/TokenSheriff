= Roles and Groups Configuration

The library provides built-in support for role-based and group-based access control through the "roles" and "groups" claims.

== Default Mapping Behavior

Both claims use `JsonCollectionMapper` by default, which detects:

* JSON arrays of strings: `"roles": ["admin", "user"]`
* JSON strings (single value): `"roles": "admin"`

It does not split space-separated or comma-separated strings, and treats nested structures as single string values.

== Supported JSON Structures

[source,json]
----
// JSON array - each element becomes a separate role/group
{ "roles": ["admin", "user", "manager"] }

// Single string - becomes a single-element list
{ "roles": "admin" }

// Mixed types - non-string values converted to strings
{ "roles": ["admin", 123, true] }
----

== Unsupported JSON Structures

[source,json]
----
// Space-separated - treated as single value "admin user manager"
{ "roles": "admin user manager" }

// Comma-separated - treated as single value "admin,user,manager"
{ "roles": "admin,user,manager" }
----

NOTE: Character-separated strings can be handled using the `StringSplitterMapper`. See <<Using StringSplitterMapper for Delimited Strings>>.

== Using Roles and Groups

[source,java]
----
List<String> roles = accessToken.getRoles();
List<String> groups = accessToken.getGroups();
boolean hasRoles = accessToken.providesRoles(List.of("admin", "user"));
boolean hasGroups = accessToken.providesGroups(List.of("group1", "group2"));
Set<String> missingRoles = accessToken.determineMissingRoles(requiredRoles);
----

== Custom Mapping for Roles and Groups

[[Using StringSplitterMapper for Delimited Strings]]
=== Using StringSplitterMapper for Delimited Strings

[source,java]
----
StringSplitterMapper commaMapper = new StringSplitterMapper(',');
StringSplitterMapper colonMapper = new StringSplitterMapper(':');

IssuerConfig issuerConfig = IssuerConfig.builder()
        .issuerIdentifier("https://issuer.com")
        .jwksContent(jwksContent)
        .claimMapper("roles", commaMapper)
        .claimMapper("groups", colonMapper)
        .build();
----

The mapper trims whitespace, omits empty values, and only works with string values.

=== Nested Claims Mapping

For nested claim structures (e.g., Keycloak's `realm_access.roles`), implement a custom `ClaimMapper`:

[source,java]
----
IssuerConfig issuerConfig = IssuerConfig.builder()
    .issuerIdentifier("https://example.com")
    .jwksContent(jwksJsonString)
    .claimMapper("roles", (mapRepresentation, claimName) -> {
        // Custom logic to extract nested roles from realm_access/roles
        // Use getNestedMap() for nested structures, getValue() for direct access
        return new JsonCollectionMapper().map(mapRepresentation, claimName);
    })
    .build();
----

=== Custom Claim Mapper Implementation

[source,java]
----
public class CustomRoleMapper implements ClaimMapper {
    @Override
    public ClaimValue map(MapRepresentation mapRepresentation, String claimName) {
        // Handle complex nested structures
        // Use getNestedMap() for nested objects, getValue()/getString() for direct access
        return new JsonCollectionMapper().map(mapRepresentation, claimName);
    }
}

IssuerConfig issuerConfig = IssuerConfig.builder()
    .issuerIdentifier("https://issuer.com")
    .jwksContent(jwksJsonString)
    .claimMapper("roles", new CustomRoleMapper())
    .build();
----

== CDI-based Custom Mapper Registration (Quarkus Only)

NOTE: This section describes Quarkus-specific functionality. The `DiscoverableClaimMapper` interface is part of the `oauth-sheriff-quarkus` module, not `oauth-sheriff-core`. For standalone core library usage, use the `ClaimMapper` interface directly via `IssuerConfig.builder().claimMapper()`.

Custom claim mappers can be registered as CDI beans and are automatically discovered and applied to all configured issuers.

=== Creating a Custom DiscoverableClaimMapper

[source,java]
----
@ApplicationScoped
public class MyCustomClaimMapper implements DiscoverableClaimMapper {

    @Override
    public String getClaimName() {
        return "custom-roles";
    }

    @Override
    public ClaimMapper getMapper() {
        return new MyCustomMapperImpl();
    }

    @Override
    public boolean isEnabled() {
        return true; // Conditionally enable via configuration
    }
}
----

=== Built-in Keycloak Mappers

[source,properties]
----
# Enable Keycloak default roles mapper (maps realm_access.roles to roles)
sheriff.oauth.keycloak.default-roles-mapper.enabled=true

# Enable Keycloak default groups mapper (processes groups claim)
sheriff.oauth.keycloak.default-groups-mapper.enabled=true
----

Both default to `false`. When enabled, the mapper is applied to all configured issuers.

=== Constraints

* Each claim name must have at most one enabled `DiscoverableClaimMapper` -- duplicates cause `IllegalStateException` at startup.
* Discovered mappers are applied globally. For issuer-specific logic, implement conditional behavior within the `ClaimMapper`.
