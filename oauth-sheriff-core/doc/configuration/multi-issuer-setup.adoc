= Multi-Issuer Configuration
:toc: left
:toclevels: 3
:source-highlighter: highlight.js
:toc-title: Table of Contents
:sectnums:


The library supports validating tokens from multiple identity providers simultaneously, with automatic issuer detection and per-issuer configuration.

== Basic Multi-Issuer Setup

[source,java]
----
TokenValidator tokenValidator = TokenValidator.builder()
    .issuerConfig(IssuerConfig.builder()
        .issuerIdentifier("https://keycloak.example.com/auth/realms/main")
        .httpJwksLoaderConfig(HttpJwksLoaderConfig.builder()
            .jwksUrl("https://keycloak.example.com/auth/realms/main/protocol/openid-connect/certs")
            .issuerIdentifier("https://keycloak.example.com/auth/realms/main")
            .build())
        .expectedAudience("my-app")
        .build())
    .issuerConfig(IssuerConfig.builder()
        .issuerIdentifier("https://auth0.example.com/")
        .httpJwksLoaderConfig(HttpJwksLoaderConfig.builder()
            .jwksUrl("https://auth0.example.com/.well-known/jwks.json")
            .issuerIdentifier("https://auth0.example.com/")
            .build())
        .expectedAudience("my-app-auth0")
        .build())
    .build();

// Validator automatically selects correct issuer based on token's "iss" claim
AccessTokenContent token = tokenValidator.createAccessToken(AccessTokenRequest.of(jwtString));
----

== Issuer Detection

The library automatically detects the issuer from the token's `iss` claim, selects the corresponding `IssuerConfig`, and validates using that issuer's configuration. If no matching issuer is found, validation fails.

=== Different Claim Mappings

[source,java]
----
// Keycloak uses nested roles - use custom ClaimMapper
ClaimMapper keycloakRolesMapper = (mapRepresentation, claimName) -> {
    // Custom logic to extract from realm_access/roles
    // Use getNestedMap() for nested structures, getValue() for direct access
    return new JsonCollectionMapper().map(mapRepresentation, claimName);
};

TokenValidator tokenValidator = TokenValidator.builder()
    .issuerConfig(IssuerConfig.builder()
        .issuerIdentifier("https://keycloak.example.com/auth/realms/main")
        .httpJwksLoaderConfig(HttpJwksLoaderConfig.builder()
            .wellKnownUrl("https://keycloak.example.com/auth/realms/main/.well-known/openid-configuration")
            .build())
        .claimMapper("roles", keycloakRolesMapper)
        .build())
    .issuerConfig(IssuerConfig.builder()
        .issuerIdentifier("https://auth0.example.com/")
        .httpJwksLoaderConfig(HttpJwksLoaderConfig.builder()

            .build())
        .build())
    .issuerConfig(IssuerConfig.builder()

== Dynamic Issuer Loading

[source,java]
----
public class DynamicTokenValidator {
    private volatile TokenValidator validator;

    public void reloadIssuers() {
        TokenValidator.Builder builder = TokenValidator.builder();
        List<IssuerConfig> issuers = loadIssuersFromDatabase();
        for (IssuerConfig issuer : issuers) {
            builder.issuerConfig(issuer);
        }
        this.validator = builder.build();
    }

    public AccessTokenContent validate(String token) throws TokenValidationException {
        return validator.createAccessToken(AccessTokenRequest.of(token));
    }
}
----

== Best Practices

* Use full URLs with protocol as issuer identifiers (`https://issuer.example.com`) -> IDP-Config
* Include realm/tenant for multi-tenant providers (`/auth/realms/main`)
* Never use wildcard or fallback issuers in production
* Validate audience claims for each issuer
* Enable `expectedTokenType("at+jwt")` for RFC 9068-compliant IdPs (see link:security-settings.adoc#token-type-validation-rfc-9068[Token Type Validation])
* Configure appropriate JWKS refresh intervals
* Use access token caching for high-traffic applications

== Troubleshooting

=== Issuer Mismatch

If tokens are rejected with "unknown issuer": check the exact `iss` claim value (including trailing slashes) and verify the issuer is added to `TokenValidator`.

=== Key Resolution Failures

If signature validation fails: verify JWKS endpoint accessibility, check for key rotation timing issues, and validate the JWKS format.

=== Claim Mapping Issues

If roles/groups are not extracted: inspect the actual token claims structure, verify claim paths match, and check if custom mappers are needed.
