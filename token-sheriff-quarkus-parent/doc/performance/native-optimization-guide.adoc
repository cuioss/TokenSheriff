== Purpose

Technical configuration for optimizing Quarkus native images for JWT validation workloads -- compiler settings, runtime parameters, and build optimizations for production deployment.

== Build Configuration

=== Compiler Optimization Settings

[source,properties]
----
# application.properties
quarkus.native.monitoring=jfr
quarkus.native.additional-build-args=--enable-url-protocols=https,--enable-http,--enable-https,-O2
----

=== GraalVM/Mandrel 25 Features

* **JCMD Support**: Native images support `jcmd` for runtime diagnostics
* **WP-SCCP**: Whole-Program Sparse Conditional Constant Propagation (default)
* **ML Profile Inference**: 1-3% performance boost via GraalNN
* **New JFR Events**: `jdk.ObjectAllocationSample` for memory profiling

[source,properties]
----
quarkus.native.monitoring=jcmd,jfr,heapdump
----

=== Reflection Configuration

The Token-Sheriff Quarkus deployment automatically registers types for reflection across several build steps, covering JWT validation pipeline, JWKS loading, token content, claim mappers, and security/algorithm classes. The total number of registered types (including transitive Quarkus framework types) varies by build.

=== Garbage Collection

[source,bash]
----
# Serial GC is the default for native images and optimal for JWT validation workloads.
# No additional build args needed.

# G1GC: Oracle GraalVM only, Linux only
native-image --gc=G1 YourApp
----

== Runtime Configuration

=== Virtual Threads

[source,properties]
----
quarkus.virtual-threads.name-prefix=jwt-validation
quarkus.virtual-threads.shutdown-timeout=10s
----

[source,java]
----
@RunOnVirtualThread
public class JwtValidationEndpoint { }
----

=== JFR Monitoring (Dual-Image Maven Profiles)

[source,xml]
----
<!-- Production build (distroless) -->
<profile>
    <id>integration-tests</id>
    <properties>
        <quarkus.native.container-build>true</quarkus.native.container-build>
        <quarkus.native.enabled>true</quarkus.native.enabled>
    </properties>
</profile>

<!-- JFR profiling build (UBI-based) -->
<profile>
    <id>jfr</id>
    <properties>
        <quarkus.native.enabled>true</quarkus.native.enabled>
        <quarkus.native.monitoring>jfr</quarkus.native.monitoring>
        <quarkus.jfr.enabled>true</quarkus.jfr.enabled>
    </properties>
</profile>
----

== Complete Configuration Examples

=== Production (Example)

NOTE: These are example configurations for reference. The actual project configuration is in `application.properties`.

[source,properties]
----
quarkus.native.monitoring=jfr
quarkus.native.additional-build-args=--enable-url-protocols=https,--enable-http,--enable-https,-O2
quarkus.virtual-threads.name-prefix=jwt-validation
quarkus.native.resources.includes=**/*.p12,**/*.crt,**/*.key
----

=== Development (Example)

[source,properties]
----
quarkus.native.additional-build-args=-O2,--enable-monitoring=jfr,heapdump
----

== Build Commands

[source,bash]
----
# Build token-sheriff-validation first (run from repository root)
./mvnw clean install -pl token-sheriff-validation

# Production build
./mvnw clean install -pl token-sheriff-quarkus-parent/token-sheriff-quarkus-integration-tests -Pintegration-tests

# JFR profiling build
cd token-sheriff-quarkus-parent/token-sheriff-quarkus-integration-tests
../../mvnw clean verify -Pjfr -Dquarkus.native.container-build=true

# Run with JFR
docker compose -f docker-compose.yml -f docker-compose.jfr.yml up -d
----

== Troubleshooting

* **Missing Reflection**: Ensure token-sheriff-validation is built before native image
* **Memory Issues**: Increase native-image-xmx to 8g or higher
* **Performance Issues**: Verify virtual threads enabled; check GC configuration; validate container resource limits

**Regression Thresholds**: >10% throughput loss, >20% latency increase, >50% memory increase.
