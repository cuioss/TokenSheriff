All `sheriff.token.*` properties for the Quarkus extension. For the core library builder API, see xref:../../../../../../token-sheriff-validation/doc/api-reference.adoc[Core API Reference].

== Minimal Configuration

[source,properties]
----
# Issuers must be explicitly enabled (default is disabled)
sheriff.token.issuers.keycloak.enabled=true
# OpenID Connect Discovery (recommended) — issuer, JWKS, and algorithms auto-discovered
sheriff.token.issuers.keycloak.jwks.http.well-known-url=https://keycloak.example.com/realms/master/.well-known/openid-configuration
----

== Issuer Configuration

Properties under `sheriff.token.issuers.<name>.*` — one block per issuer.

[cols="2,1,1,3", options="header"]
|===
|Property |Type |Default |Description
|`enabled` |boolean |`false` |Enable/disable this issuer. Must be explicitly set to `true` to activate the issuer.
|`issuer-identifier` |String |— |Matched against `iss` claim. Auto-discovered when using well-known.
|`expected-audience` |String (csv) |— |Expected `aud` claim values
|`expected-client-id` |String (csv) |— |Expected `azp` / `client_id` claim values
|`algorithm-preferences` |String (csv) |— |Allowed signature algorithms (default: all supported)
|`claim-sub-optional` |boolean |`false` |Allow missing `sub` claim
|`expected-token-type` |String |— (disabled) |Expected `typ` header value. Set to `at+jwt` for RFC 9068-compliant IdPs. Most IdPs (Keycloak, Zitadel, Dex) do *not* set `at+jwt` — verify before enabling.
|`audience-validation-disabled` |boolean |`false` |Disable audience validation for this issuer
|`access-token-audience-optional` |boolean |`false` |Allow access tokens without an `aud` claim
|`clock-skew-seconds` |int |`60` |Clock skew tolerance for `exp` / `nbf` validation
|`max-token-age-seconds` |int |— (disabled) |Max token age based on `iat`; rejects when `now - iat > max-token-age-seconds + clock-skew-seconds`
|===

== Bearer Token Error Responses

Error responses created via `BearerTokenResult.createErrorResponse()` follow RFC 6750
(OAuth 2.0 Bearer Token Usage) with one deliberate deviation:

[cols="2,1,3", options="header"]
|===
|Condition |Status |`WWW-Authenticate` error code
|No token present |`401` |—
|Invalid or expired token |`401` |`invalid_token`
|Malformed bearer token (e.g. empty) |`400` |`invalid_request`
|Missing required scopes |`401` |`insufficient_scope` (deviation, see below)
|Missing required roles/groups |`403` |`insufficient_privileges`
|===

*Deviation from RFC 6750:* RFC 6750 Section 3.1 specifies `403 Forbidden` for
`insufficient_scope`. This extension deliberately returns `401 Unauthorized` instead,
following the OAuth 2.0 Step-Up Authentication Challenge protocol (RFC 9470): an
insufficient-scope challenge is actionable — the client can re-authenticate or obtain a
token with higher scope/ACR and retry. Role/group violations are terminal (no
re-authentication can fix them) and therefore return `403`. When both scope and
role/group requirements are unmet, the scope challenge (`401`) takes precedence so the
client gets a chance to step up first.
