== Purpose

JWT validation performance characteristics, baselines, and analysis for the CUI OAuth Sheriff Quarkus integration.

== Performance Baselines (March 2026)

=== System Configuration

* **Platform**: Apple M4, 10 CPU cores, 32 GB RAM
* **Runtime**: Mandrel jdk-25 native image
* **Framework**: Quarkus 3.34.0
* **Container**: Docker with Keycloak for token issuance

=== Current Performance Summary

[cols="3,2,2,2", options="header"]
|===
| Endpoint | Throughput | Avg Latency | P99 Latency

| **JWT Validation** | 25,178 ops/s | 2.30ms | 11.52ms
| **Health Check** | 98,211 ops/s | 0.78ms | 6.04ms
|===

All measurements on Mandrel native image at 50 concurrent connections (optimal). See link:../../../benchmarking/doc/Analysis-03.2026-Integration.adoc[Integration Analysis] for connection sweep (50-300).

== Latency Decomposition

From the link:../../../benchmarking/doc/Analysis-03.2026-Latency-Decomposition.adoc[ablation study]:

[cols="3,2,2", options="header"]
|===
| Layer | Latency (avg) | % of total

| Shared infrastructure (Docker, TLS, HTTP/2, Quarkus routing) | 0.78ms | 33.9%
| JAX-RS POST routing + Jackson serialization | 0.40ms | 17.4%
| Vert.x context resolution | 0.06ms | 2.6%
| Authorization parsing + HashMap construction | ~0ms | ~0%
| **JWT validation under concurrency** | **1.06ms** | **46.1%**
| CDI proxy + `@Timed` | ~0ms | ~0%
|===

**Total**: 0.78 + 0.40 + 0.06 + 0 + 1.06 + ~0 ≈ **2.30ms**

== Root Cause: CPU Saturation

RSA signature verification is compute-bound (88% of the validation pipeline). With 114 avg threads competing for 10 CPU cores at 85% peak utilization, the system is CPU-saturated (~11:1 thread-to-core oversubscription).

=== JMH Concurrency Sweep (HotSpot JVM)

A concurrency sweep on HotSpot JVM (OpenJDK 21.0.7, Apple M4, 10 cores) confirms this is **not native-image-specific**:

[cols="1,2,2,2", options="header"]
|===
| Threads | Avg Latency | Multiplier vs 1 thread | Interpretation

| 1 | 48µs | 1.0x | Baseline — single core
| 10 | 89µs | 1.9x | Fits 10 cores — nearly linear
| 50 | 556µs | 11.6x | 5:1 oversubscription
| 100 | 957µs | 20.0x | 10:1 oversubscription
|===

The native integration cost of 1,060µs at 50 connections adds ~2x on top of the JVM baseline (556µs) — consistent with HTTP/TLS/Quarkus overhead, not a native-image penalty.

See link:../../../benchmarking/doc/Analysis-03.2026-Integration.adoc#_root_cause_cpu_saturation_from_rsa_under_concurrency[Integration Analysis root cause] for full details.

== JWT Validation Pipeline

RSA signature verification dominates the validation pipeline (88% at single-thread):

[cols="2,2,2", options="header"]
|===

| RSA Signature Verification | 68µs | **88%**

| Other | ~4.6µs | 6%
|===

== Algorithm Performance

[cols="2,2,2,3", options="header"]
|===
| Algorithm | Type | Security Level | Notes

| RS256/384/512 | RSA | 2048-bit | 48µs (1t) / 556µs (50t) / 1,060µs (native 50c) — current default
| ES256/384/512 | ECDSA | 256-384-bit curves | Not benchmarked
| Ed25519/Ed448 | EdDSA | Curve25519/448 | Not benchmarked
|===

No alternative algorithm has been benchmarked. Algorithm choice should be driven by standards compliance, token size, and key management.

== JCA Provider Landscape

* **Default JDK** (Recommended): Full native image support, automatic updates
* **BouncyCastle**: Only if specific algorithms needed; requires reflection configuration
* **Conscrypt**: Not suitable for GraalVM native image (BoringSSL dependencies)
* **Jipher**: No longer bundled with GraalVM 25; consider only for FIPS compliance

Concurrency scaling is a JVM-level characteristic, not provider-specific.

== Benchmark Execution

[source,bash]
----
# Integration benchmarks (native image)
cd oauth-sheriff-quarkus-parent/oauth-sheriff-quarkus-integration-tests
../../mvnw clean verify -Pintegration-tests

# JMH microbenchmarks (HotSpot JVM)
cd benchmarking/benchmark-core
../../mvnw clean verify -Pbenchmark

# Concurrency sweep (1, 10, 50, 100 threads on HotSpot JVM)
cd benchmarking/benchmark-core
../../mvnw clean verify -Pbenchmark-concurrency-sweep

# JFR profiling
docker compose -f docker-compose.yml -f docker-compose.jfr.yml up -d
----

**Regression Thresholds**: >10% throughput loss, >20% latency increase, >50% memory increase.
