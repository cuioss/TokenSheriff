= Code Findings: Compatibility vs. Correct Design
:toc: left
:toclevels: 3
:sectnums:

== Overview

Semantic analysis of all production code asking one question per artifact:
_"Does this exist because it is the right design, or because we are afraid to break something?"_

Maturity: *pre-1.0* — Breaking changes are free. No deprecation, no legacy support, no transitional code. Recommend removing/fixing everything.

Date: 2026-04-03

Modules analyzed:

* `oauth-sheriff-core` — 65 production files
* `oauth-sheriff-quarkus` — 37 production files

Excluded: benchmarking (tooling), deployment (build-time), integration-tests (test scaffolding).

== Critical (Security Impact)

No critical findings. The codebase follows fail-closed patterns throughout.

== High (API / Design)

[cols="1,3,4,3"]
|===
|ID |Location |Issue |Recommendation

|H-1
|`JWTValidationLogMessages.java:559-562`
|`DPOP_HTU_HTM_SKIPPED` LogRecord (identifier 170) is defined but never referenced anywhere in production code. A WARN-level log message was intended for `DpopProofValidator` when htu/htm validation is skipped, but the call site was never wired.
|Wire into `DpopProofValidator` where htu/htm validation is skipped, or remove the unused constant. Unused log records create noise in `LogMessages.adoc` documentation.

|H-2
|`VertxHttpServletRequestAdapter.java` (entire file, 833 lines)
|Oversized adapter: the class implements `HttpServletRequest` but no production code calls `resolveHttpServletRequest()` on `VertxServletObjectsResolver`. The Vertx implementation overrides `resolveHeaderMap()`, `resolveRequestUri()`, and `resolveRequestMethod()` — the only three methods called in production. The remaining 800 lines (sessions, cookies, locale parsing, multipart, etc.) all throw `UnsupportedOperationException`. This exists because the `HttpServletRequestResolver` interface mandates returning an `HttpServletRequest`, not because it's the right design.
|Redesign `HttpServletRequestResolver` to only declare `resolveHeaderMap()`, `resolveRequestUri()`, and `resolveRequestMethod()`. Delete `VertxHttpServletRequestAdapter` entirely. This removes 800 lines of dead adapter code.

|H-3
|`AccessLogFilterConfigProducer.java`
|Unnecessary CDI producer indirection. `AccessLogFilterConfigResolver` takes only a `Config` in its constructor and could itself be an `@ApplicationScoped` bean with `@Inject`.
|Make `AccessLogFilterConfigResolver` itself an `@ApplicationScoped` CDI bean with `@Inject` constructor. Delete `AccessLogFilterConfigProducer`.
|===

== Medium (Dead Code / Unnecessary API Surface)

[cols="1,3,4,3"]
|===
|ID |Location |Issue |Recommendation

|M-1
|`SecurityEventCounter.java:181-183`
|`reset()` was reported as having zero production callers, but `JwtMetricsCollector.clear()` in the quarkus module calls it. *False positive* — `reset()` is correctly public.
|No action needed. Finding dismissed.

|M-2
|`TokenStringValidator.java:64-68`
|Two-argument constructor is never called in production — `TokenValidator.java:305` uses only the three-argument constructor. The two-argument constructor exists for test convenience.
|Remove the two-argument constructor. Pre-1.0 — breaking changes are free. The three-argument constructor with `@Nullable JweDecryptionConfig` handles the null case.

|M-3
|`BearerTokenResult.java:142-150`
|`constraintViolation()` static factory method has zero production callers. `BearerTokenProducer` builds `CONSTRAINT_VIOLATION` results via the builder directly. Premature API surface for a pre-1.0 library.
|Remove until a consumer actually needs it.

|M-4
|`JsonWebTokenAdapter.java:142-144`
|`getTokenContent()` is public but only used in tests — zero production callers. Leaks internal `TokenContent` structure through the MP-JWT `JsonWebToken` adapter.
|Remove or make package-private.
|===

== Low (Cleanup)

[cols="1,3,4,3"]
|===
|ID |Location |Issue |Recommendation

|L-1
|`AudienceValidator.java:98-101`
|Lines 99 and 101 both log `ACCESS_TOKEN_AUDIENCE_MISSING`. The `accessTokenAudienceOptional=true` branch (line 99) logs a WARN but does not increment the security event counter, while the `false` branch (lines 101-102) increments the counter. Inconsistent instrumentation.
|Don't log WARN for expected/configured behavior (creates alert fatigue), or count the event in both branches for consistent monitoring.

|L-2
|`AccessLogFilterConfig.java:42-86`, `AccessLogFilterConfigResolver.java:66-74`
|Dual-source defaults: builder `@Builder.Default` values (e.g., `minStatusCode = 400`, `enabled = false`) AND resolver `.orElse()` fallbacks define the same defaults. If one changes without the other, behavior depends on which code path runs.
|Remove `.orElse()` fallbacks from the resolver. Builder defaults are the canonical source.

|L-3
|`AccessLogFilterConfig.java:98-114`
|Manual null-safe getters for `includePaths`, `excludePaths`, `includeStatusCodes` that shadow Lombok `@Value`-generated getters. Null checks exist from fear of uninitialized builder fields, but the resolver always provides values via the builder.
|Add `@Builder.Default` to the three `List` fields (default `List.of()`). Remove manual getters.

|L-4
|`IssuerConfigResolver.java:72-74`
|`@Inject` constructor takes only `Config` but is never used in production — `TokenValidatorProducer.java:119` always calls the 4-argument constructor. The CDI constructor suggests discoverability that doesn't exist (`IssuerConfigResolver` is manually instantiated, not CDI-managed).
|Remove the `@Inject` constructor and the corresponding CDI bean registration in `OAuthSheriffProcessor` — it's dead code that caused CDI to attempt unsatisfiable injection.

|L-5
|`ClientIpExtractor.java:82-84`
|Single-parameter `extractClientIp(headers)` overload just delegates to `extractClientIp(headers, null)`. The only production caller (`CustomAccessLogFilter.java:181`) uses the 2-argument version directly.
|Remove the single-parameter overload — zero production callers.
|===
