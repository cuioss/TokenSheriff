== Overview

Semantic analysis of all production code (not keyword search) asking one question per artifact: _"Does this exist because it is the right design, or because we are afraid to break something?"_

Pre-1.0 means: no deprecation, no legacy support, no transitional code, no backward compatibility shims. Breaking changes are free.

Date: 2026-03-27

== Critical (Security Impact)

These findings represent cases where leniency, silent fallbacks, or fail-open behavior in a security library could lead to vulnerabilities.

[cols="1,3,4,3,3"]
|===
|ID |Location |Issue |Recommendation |Security Review

|C-1
|`JwkKeyHandler:241-248`
|Unknown EC curve silently defaults to `ES256` instead of throwing. Inconsistent with `getEcParameterSpec` (line 194) which correctly throws for unsupported curves.
|Throw `IllegalArgumentException` for unknown curves.
|*Confirmed* (Medium). Also `KeyProcessor:217` defaults null curve to P-256. Attacker-supplied JWK with crafted curve name gets ES256 verification instead of rejection.

|C-2
|`JweDecryptor:315-362`
|DEFLATE decompression falls back from raw DEFLATE to zlib-wrapped. Comment says _"for compatibility (some implementations use zlib wrapper)"_. RFC 7516 specifies raw DEFLATE only.
|Remove the fallback. Reject non-compliant zlib-wrapped input.
|*Confirmed* (Low). Both paths enforce 256KB `MAX_DECOMPRESSED_SIZE` limit, mitigating compression bombs. Risk is standards non-compliance rather than practical exploit.

|C-3
|`JweDecryptionConfig:122`
|JWE compression enabled by default (`compressionEnabled = true`). Compression oracles (BREACH-style attacks) are a known risk. The infrastructure to reject compressed tokens already exists.
|Change default to `false`. Require explicit opt-in.
|*Partially confirmed* (Low). `JweDecryptor` validates zip header (only accepts `"DEF"`), enforces 256KB decompression limit. Mitigations reduce BREACH risk significantly. Default-on is marginally less secure but reasonable for JWE interoperability.

|C-5
|`TokenType:88-96`
|`fromTypeClaim()` returns `UNKNOWN` (with empty mandatory claims set) for unrecognized type strings. A token with a garbage `typ` claim bypasses all mandatory claim checks.
|Throw `TokenValidationException` for unrecognized types.
|*Partially confirmed* (Low). `UNKNOWN` has empty `mandatoryClaims` set, bypassing mandatory claims validation. However, other validators (signature, expiration, issuer) still run. Impact limited to skipping audience/subject checks for garbage `typ`.

|C-6
|`BearerTokenInterceptor:117-119`
|Security interceptor fails open: when `@BearerAuth` annotation is not found (CDI proxy edge case), it warns and calls `ctx.proceed()` -- executing without authentication.
|Fail closed: throw `IllegalStateException`.
|*Confirmed* (High). Fail-open pattern in authentication interceptor. If interceptor fires without matching annotation (misconfiguration or CDI proxy edge case), unauthenticated requests pass through. Should fail closed.

|C-7
|`IssuerConfigResolver:494-498`
|Config key name (e.g., `"keycloak"`) used as fallback `issuerIdentifier` when not explicitly configured. The config property key silently becomes the `iss` value to match against tokens.
|Require explicit `issuer-identifier` when using direct JWKS URL. Fail at startup with a clear error.
|*Confirmed* (Medium). Configuration footgun: logical config key silently becomes issuer claim to validate against. Mismatch between config key and actual token issuer could cause validation failures or bypasses.

|C-8
|`IssuerConfigResolver:209`
|Issuer defaults to enabled (`.orElse(true)`) when `enabled` property is not set. A typo in the property name silently activates an issuer with partial config.
|Default to `false`. Require explicit `enabled=true`.
|*Confirmed* (Low). Usability-oriented default. Partially configured issuers would likely fail at builder validation. Low practical risk but inconsistent with fail-safe principles.

|C-9
|`JwksParser:84-109`
|When JWKS parsing fails, silently falls back to parsing as a bare single JWK. A JWKS endpoint returning a bare JWK is non-compliant per RFC 7517 Section 5.
|Remove the bare JWK fallback, or make it opt-in via config.
|*Acknowledged -- by design* (Low). Common interoperability pattern. Parsed key validates `kty` field. Accepting broader input does not weaken cryptographic validation.
|===

== High (API / Design)

Design issues that affect the 1.0 API contract, create false security assumptions, or embed compatibility workarounds.

[cols="1,3,4,3,3"]
|===
|ID |Location |Issue |Recommendation |Security Review

|H-1
|`ParserConfig` (formerly lines 95-142)
|`maxBufferSize` field was configured, stored, documented, included in `equals`/`hashCode` -- but never enforced. DSL-JSON `Settings` has no buffer-size API. Created a false sense of security.
|*Resolved*. Field, constant, builder method, configuration property, log message parameter, and all documentation references removed.

|H-2
|`IssuerConfig:116-119`
|`expectedAudience` defaults to `Set.of()`. `AudienceValidator` (line 69) skips validation entirely when empty. Combined with H-3, this means audience validation is silently disabled by default.
|Require `expectedAudience` at build time, or provide an explicit `audienceValidationDisabled` flag.
|*Confirmed* (Medium). Empty `expectedAudience` causes `AudienceValidator` to short-circuit at line 69-72 with debug log only. Fail-open on missing configuration for a security library.

|H-3
|`TokenClaimValidator:95-103`
|Constructor warns when `expectedAudience` / `expectedClientId` are empty but proceeds. Validation is silently skipped at runtime with only a startup log warning.
|Require these at construction or force explicit opt-out.
|*Partially confirmed* (Low). Duplicate of H-2 from constructor angle. WARN log + security event counter provide operational visibility. The actual gap is in H-2's validation skip.

|H-4
|`IssuerConfig:128-147`
|`claimSubOptional` is an ad-hoc escape hatch. Javadoc says _"a workaround for identity providers that don't include the subject claim."_ Will accumulate siblings (`claimIatOptional`, etc.) over time.
|Replace with a configurable `Set<ClaimName> optionalClaims` per issuer.
|*Acknowledged -- by design* (Low). Defaults to `false` (secure). Javadoc warns about RFC 7519 compliance relaxation. Pragmatic accommodation for real-world IdPs like Keycloak. Dedicated test class exists.

|H-5
|`TokenContent:57`
|Core interface extends `org.eclipse.microprofile.jwt.JsonWebToken`. Drags MP-JWT coupling into core: `upn` claim, `Principal`, epoch-second APIs. The `getName()` fallback chain (lines 96-104) is MP-JWT spec, not the library's concern.
|Move `JsonWebToken` to a Quarkus-layer adapter. Keep core free of MP-JWT dependency.
|*Acknowledged -- by design* (Informational). Intentional architectural decision documented in Javadoc. Provides `Principal` integration and MP-JWT injection compatibility. Design trade-off, not a security vulnerability.

|H-6
|`IssuerConfig:249-258` + `IssuerConfigCache:102-107`
|Double JWKS initialization. `initSecurityEventCounter` calls `initJWKSLoader().join()` (blocking). Then `IssuerConfigCache` calls `initJWKSLoader()` again (async). Comment says _"For now, we block on completion -- proper async handling will be in C6"_.
|Remove the blocking init. Use a single async init path.
|*Confirmed* (Low). `HttpJwksLoader.initJWKSLoader` lacks idempotency guard -- second call triggers redundant HTTP request and potential race condition. Real code defect but low security impact.

|H-7
|`VertxHttpServletRequestAdapter`
|826-line adapter implementing full `HttpServletRequest`. 27 methods throw `UnsupportedOperationException`. Production usage is headers only (`getHeaderNames()`, `getHeaders()`).
|Change core API to accept a header map or minimal interface instead of `HttpServletRequest`.
|*Partially confirmed* (Low). Code quality / maintainability concern. Adapter is internal to Quarkus module, not exposed to application code. Unsupported methods throw with descriptive messages. Dedicated test exists.
|===

== Medium (Dead Code / Unnecessary API Surface)

Code that is unused in production, structurally questionable, or adds API surface without justification.

[cols="1,3,4,3,3"]
|===
|ID |Location |Issue |Recommendation |Security Review

|M-1
|`ParserConfig:210-238`
|Builder accepts zero and negative size limits. No validation. Other builders in the codebase (e.g., `IssuerConfig`) validate inputs.
|Add `Preconditions.checkArgument(value > 0)` to each setter.
|*Confirmed* (Medium). Missing validation on security-critical size limits. `maxTokenSize(0)` would disable token-size protection. Add positive-value validation to constructor.

|M-2
|`DecodedJwt:73-84`
|Null-tolerant getters return synthetic fallback objects (blank `JwtHeader`, empty `MapRepresentation`) instead of failing. Record fields are always set by `NonValidatingJwtParser`, so null is a bug.
|Validate non-null in constructor. Remove null-coalescing getters.
|*Acknowledged* (Low). Null-object pattern simplifies consumer code. `NonValidatingJwtParser` sets fields; callers check `body.isEmpty()`. Not a security risk in practice.

|M-3
|`NonValidatingJwtParser:203-249`
|Three public `decode` methods including `decode(String, boolean)` -- opaque boolean parameter. Only caller passes `false`.
|Keep `decode(String)` and `decodeOpaqueToken(String)`. Remove the boolean overload.
|*Partially confirmed* (Low). Boolean parameter smell. Each method serves distinct purpose. Could make boolean overload package-private. No security vulnerability.

|M-4
|`SecurityEventCounter:114-117`
|Success event types (`ACCESS_TOKEN_CREATED`, etc.) have `null` LogRecord and `null` Category. Calling `getId()` or `getDescription()` on these throws NPE.
|Either give success types proper metadata or separate them into a different type.
|*Acknowledged* (Informational). Intentional per CUI logging standards: only INFO/WARN/ERROR need LogRecords. Success events are metric counters only. Null values are documented.

|M-5
|`SignatureAlgorithmPreferences:109-118`
|`getMostPreferredAlgorithm()` has zero production callers. Only called in tests.
|Remove it.
|*Confirmed* (Informational). Dead code. No security impact.

|M-6
|`WellKnownResult:77-93`
|`EMPTY` sentinel using `"about:empty"` magic strings with a three-way `isEmpty()` check (identity, magic string, null fields). Never referenced from production code outside the class.
|Use `Optional.empty()` instead.
|*Partially confirmed* (Low). Magic string approach is fragile. `HttpWellKnownResolver` uses Optional-returning methods, limiting propagation risk. Low actual risk.

|M-7
|`WellKnownResult:179-193`
|`supportsFullOAuthFlows()` and `isMinimal()` -- zero production callers. Only called in tests.
|Remove them. The library validates JWTs; it does not initiate OAuth flows.
|*Confirmed* (Informational). Dead code. No security impact.

|M-8
|`HttpWellKnownResolver:122-155`
|Exposes `getAuthorizationEndpoint()`, `getTokenEndpoint()`, `getUserinfoEndpoint()`. Never called in production. Only called in tests.
|Remove them. `getWellKnownResult()` provides access to the full document if needed.
|*Partially confirmed* (Informational). Reasonable library API surface for consumers. Not used internally but may serve external integrators. No security impact.

|M-10
|`BearerTokenResult:116-130`
|`success()` static factory method is unused in production. `BearerTokenProducer` builds success results directly via the builder.
|Remove or make it the canonical success path.
|*Confirmed* (Informational). Unused factory method. No security impact.

|M-11
|`IssuerConfigResolver:75-125`
|Four telescoping constructors. Only the 4-arg constructor is used in production (`TokenValidatorProducer:119`). The 1-, 2-, and 3-arg constructors are test-only.
|Keep only the 4-arg constructor. Tests can use it directly.
|*Partially confirmed* (Informational). Standard telescoping pattern. Simpler constructors delegate to 4-arg with null defaults. No security concern.

|M-12
|`NonValidatingJwtParser:151`
|Constructor silently replaces null `ParserConfig` with default. The builder already initializes the field. Belt-and-suspenders pattern hides programming errors.
|Require non-null in constructor.
|*Confirmed* (Low). Constructor is private; null only arrives via `builder().config(null).build()`. Silent replacement could mask misconfigured security limits. Use `Objects.requireNonNull`.

|M-13
|`TokenBuilder:84-93`
|`createAccessToken()` returns `Optional.empty()` when body is empty. At this pipeline stage (after parsing, issuer extraction, signature verification), an empty body is an impossible state. Same for `createIdToken` (line 101).
|Throw `TokenValidationException` instead of returning `Optional.empty()`.
|*Acknowledged* (Informational). Caller in `AccessTokenValidationPipeline` handles empty Optional. Standard Optional usage, not dead code.
|===

== Low (Cleanup)

Minor issues worth addressing before 1.0 to avoid permanent API commitments.

[cols="1,3,4,3,3"]
|===
|ID |Location |Issue |Recommendation |Security Review

|L-1
|`IdentityMapper:43-48`
|Silently `toString()`s non-String claim values (numbers, booleans, nested objects). A malformed token with a JSON object as `iss` becomes `{foo=bar}`.
|Reject unexpected types or log a warning.
|*Acknowledged* (Informational). Documented behavior for an "identity mapper." Complex objects produce ugly but non-exploitable toString output.

|L-2
|`CollectionClaimHandler:84-88`
|`providesValues(null/empty)` returns `true` (vacuous truth). Calling code with a misconfigured empty required-scopes list silently passes.
|Throw `IllegalArgumentException` on null. Document empty-set behavior.
|*Confirmed* (Low). Empty-set-is-subset semantics are mathematically correct but could mask misconfiguration. Callers use `orElse(false)` for absent claims, limiting practical risk.

|L-3
|`KeycloakDefaultGroupsMapper:64-78`
|Ignores `claimName` parameter; always reads hardcoded `"groups"` claim. Same for `KeycloakDefaultRolesMapper` (hardcodes `"realm_access"`).
|Either make the interface contract explicit about fixed-source mappers or remove the unused parameter.
|*Confirmed* (Low). `claimName` parameter is accepted but unused -- only appears in debug log. Violates interface contract. Could silently read wrong data if mapper configured for non-"groups" claim.

|L-4
|`AccessTokenContent:157-162`
|Dual-source `getEmail()`: constructor field takes precedence, then falls back to claims map. Two sources of truth for the same field.
|Use claims map as the single source.
|*Confirmed* (Low). Constructor-provided email takes priority over claim email. Could produce inconsistent values in audit trails. Minor design concern.

|L-5
|`ServletObjectsResolver:41-63`
|CDI qualifier with a `Variant` enum containing exactly one value: `VERTX`. Every injection site must specify `@ServletObjectsResolver(Variant.VERTX)` for zero benefit.
|Remove the enum. Use a simple qualifier or inject directly.
|*Confirmed* (Informational). Over-engineered for single implementation. No security impact.

|L-6
|`JwksHttpContentConverter:54-56`
|No-arg constructor creates a default `ParserConfig`. Only used in one test. Production always passes explicit config.
|Remove the no-arg constructor.
|*Confirmed* (Informational). Test-only convenience constructor. No security impact.

|L-7
|`TokenValidatorHealthCheck:54`
|Null-checks CDI-injected `issuerConfigs` field. CDI constructor injection guarantees non-null or fails at startup.
|Remove the null check. Keep only `isEmpty()`.
|*Partially confirmed* (Informational). Defensive null guard is harmless. CDI guarantees non-null for constructor injection but guard protects against edge cases. No security issue.

|L-8
|`BearerTokenProducer:137-139`
|Public `getAccessTokenContent()` discards all error information from `BearerTokenResult`. Callers cannot distinguish "no token" from "expired" from "invalid signature".
|Make it private or change return type to `BearerTokenResult`.
|*Confirmed* (Low). Usability concern that could lead to poor error handling in consuming code. `getBearerTokenResult()` is available as richer alternative.

|L-9
|`ClaimName:166-174`
|`UPN` claim is a MicroProfile JWT Auth invention, not an RFC or OIDC claim. Only consumer is `TokenContent.getName()` (MP-JWT principal fallback chain).
|Move to Quarkus layer together with H-5.
|*Confirmed* (Informational). MP-JWT specific claim, properly documented. Necessary for MP-JWT compatibility. No security concern.

|L-10
|`MandatoryClaimsValidator:100-113`
|`shouldSkipClaimValidation` is a point-fix for `sub` only. Related to H-4 (`claimSubOptional`).
|Will be resolved by implementing H-4's configurable optional claims set.
|*Confirmed* (Low). Generic method name for `sub`-only logic. Current implementation correct for stated purpose. Resolves with H-4.

|L-11
|`AccessTokenValidationPipeline:204-249`
|Inconsistent null guards on map lookups. `signatureValidators.get()` has a null check (line 214); `headerValidators.get()` and `claimValidators.get()` do not -- would NPE.
|Either validate all map lookups consistently or guarantee non-null at construction time.
|*Confirmed* (Medium). Three of four map lookups lack null guards, would throw unhandled NPE on initialization bugs. Inconsistency means some failures are clear, others opaque. Warrants consistent guards.

|L-12
|`SecurityEventCounter:211-213`
|`reset(EventType)` overload is only called in tests. The no-arg `reset()` is used in production (`JwtMetricsCollector`).
|Remove the `reset(EventType)` overload.
|*Partially confirmed* (Informational). Test-only utility method. Natural API for a counter class. Minor cleanliness issue, not security concern.
|===
