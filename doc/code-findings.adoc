= Code Findings: Compatibility vs. Correct Design
:toc: left
:toclevels: 3
:sectnums:

== Overview

Semantic analysis of all production code asking one question per artifact:
_"Does this exist because it is the right design, or because we are afraid to break something?"_

Maturity: **pre-1.0** — Breaking changes are free. No deprecation, no legacy support, no transitional code.

Date: 2026-04-02

Scope: `oauth-sheriff-core`, `oauth-sheriff-quarkus` (all production code, excluding benchmarking/BOM/test code)

== Critical (Security Impact)

No critical findings. The codebase is consistently fail-closed in all main validation paths.

DPoP validation was specifically audited: `DpopProofValidator.validateHtuHtm()` correctly throws `TokenValidationException` when `requestUri` or `requestMethod` is null/blank (lines 342-349), preventing any fail-open when Vert.x context resolution fails in `BearerTokenProducer.resolveRequestContext()`.

== High (API / Design)

No high findings after verification. The three original candidates were downgraded:

* `AudienceValidator` empty-set skip: By design — `TokenClaimValidator` constructor logs WARN (line 96-99), and the validator is always constructed through that path. Moved to Medium for the public visibility concern.
* `SignatureTemplateManager` uncached algorithm fallback: `TokenHeaderValidator.validateAlgorithm()` rejects non-preferred algorithms *before* `SignatureTemplateManager` is called. The fallback only accepts the 10 known algorithms (switch default throws `IllegalArgumentException`). Moved to Medium as defense-in-depth.
* DPoP context fail-open: False positive — core validator handles nulls correctly.

== Medium (API / Design / Missing Validation)

[cols="1,3,4,3"]
|===
|ID |Location |Issue |Recommendation

|M-1
|`IdTokenValidationPipeline:139-169`
|**Missing null-checks on map lookups.** Unlike `AccessTokenValidationPipeline` (which checks for null and throws `IllegalStateException` at lines 207-209, 219-221, 232-234, 258-260), the ID token pipeline directly dereferences `headerValidators.get()`, `signatureValidators.get()`, `tokenBuilders.get()`, and `claimValidators.get()` without null checks. NPE on misconfiguration instead of a clear error.
|Add the same null-check pattern as `AccessTokenValidationPipeline`, or extract a shared lookup helper.

|M-2
|`AudienceValidator:98-99`
|**`accessTokenAudienceOptional` bypass only logged at DEBUG.** When an access token omits the `aud` claim and `accessTokenAudienceOptional=true`, the confused-deputy risk is logged only at DEBUG level — invisible in production.
|Log at WARN level when this path is taken, since it represents an active security relaxation per request.

|M-3
|`AudienceValidator:51` (class visibility)
|**Internal class is public.** `AudienceValidator` is annotated `@apiNote ... internal ... not part of the public API` but the class is `public`. It can be independently constructed with an empty `expectedAudience`, which silently skips all audience validation (line 71-73).
|Make the class package-private to match its documented intent.

|M-4
|`SignatureTemplateManager:114-122`
|**Defense-in-depth: algorithm allowlist not enforced at crypto layer.** When an algorithm is not pre-cached, `getSignatureInstance()` creates a new template on-the-fly. `TokenHeaderValidator` validates the algorithm first, so this path is only reachable if a future code change bypasses header validation.
|Throw `IllegalArgumentException` if the algorithm is not in the pre-cached set. The `default` case in `createSignatureTemplate` already rejects unknown algorithms, but non-preferred known algorithms would still get templates.

|M-5
|`NonValidatingJwtParser` builder `:162-185`
|**Builder allows construction without `SecurityEventCounter`.** The field defaults to `null`, causing NPE at parse time (e.g., `securityEventCounter.increment()` in `decodeInternal()`). Production always provides one via `TokenValidator`, but the builder API does not enforce it.
|Add `Objects.requireNonNull(securityEventCounter)` in `build()`.

|M-6
|`ValidationContext:77-81`
|**No validation on `clockSkewSeconds`.** Accepts negative values, which would cause `expirationTime.plusSeconds(negative)` to shrink the tolerance window, making valid tokens appear expired. The `IssuerConfig` defaults to 60 but `ValidationContext` constructors do not validate.
|Add `if (clockSkewSeconds < 0) throw new IllegalArgumentException(...)` in both constructors.

|M-7
|`AccessTokenCacheConfig:74-82`
|**No validation on `evictionIntervalSeconds`.** Accepts zero or negative values. Zero schedules eviction at maximum frequency; negative throws at `scheduleWithFixedDelay`. `@Builder.Default` is 10, but the builder allows any value.
|Add validation: `evictionIntervalSeconds` must be positive when `maxSize > 0`.

|M-8
|`ClaimValue:99-105`
|**Public constructor allows inconsistent state.** The constructor accepts any combination of fields, enabling invalid states (e.g., `STRING` type with non-null `dateTime`, or `DATETIME` type with a non-empty list). Three well-typed static factories exist (`forPlainString`, `forList`, `forDateTime`).
|Make the constructor package-private. Force all callers through the static factories.

|M-9
|`VertxHttpServletRequestAdapter` (833 lines)
|**Oversized adapter — production bypasses it entirely.** Implements full `HttpServletRequest` but `VertxServletObjectsResolver` overrides `resolveHeaderMap()`, `resolveRequestUri()`, and `resolveRequestMethod()` with direct Vert.x calls. The adapter is only created in `resolveHttpServletRequest()`, which is never called in production. Its own Javadoc acknowledges this (line 41-44).
|Evaluate whether `HttpServletRequestResolver.resolveHttpServletRequest()` can be removed from the interface. If kept for extensibility, make the adapter package-private and document as internal-only.

|M-10
|`DecryptionKeyLoader:150-157`
|**Silently swallows RSA exception.** Tries RSA first, catches `GeneralSecurityException ignored`, then tries EC. If the key IS RSA but has corrupted data, the error misleadingly says "not RSA or EC." The RSA-specific root cause is lost.
|Store the RSA exception and attach it as a suppressed exception on the final EC `GeneralSecurityException`.

|M-11
|`ClientIpExtractor:59-98`
|**Trusts all proxy headers without verification.** Any client can send `X-Forwarded-For` or `CF-Connecting-IP` headers. Extracted IP is used in `CustomAccessLogFilter` for access logging. The Javadoc (lines 36-48) warns about this but the code has no enforcement mechanism. Impact limited to log injection — not used for security decisions.
|Document that this class must only be used behind a configured reverse proxy. Consider falling back to Vert.x `remoteAddress()` when `quarkus.http.proxy.proxy-address-forwarding` is false.
|===

== Low (Cleanup)

[cols="1,3,4,3"]
|===
|ID |Location |Issue |Recommendation

|L-1
|`DpopProofValidator:307-309`
|**Hardcoded 60s future tolerance.** DPoP proof freshness uses `age < -60` instead of the per-issuer `clockSkewSeconds`. The rest of the pipeline uses configurable clock skew consistently.
|Use the issuer's `clockSkewSeconds` instead of the hardcoded `-60`.

|L-2
|`DecodedJwt` compact constructor `:75-84`
|**Does not validate `header` is non-null.** Validates `body` but not `header`. A null header silently returns `Optional.empty()` for `getKid()`/`getAlg()` instead of failing as structurally invalid. The `DecodedJwt.of()` factory (line 97-99) does check, but the direct constructor does not.
|Add `Objects.requireNonNull(header, "header must not be null")` in the compact constructor.

|L-3
|`TokenContent.containsClaim(String)` `:118-120`
|**Zero callers anywhere.** Default interface method not used in production or test code. Speculative API surface on the 1.0 public interface.
|Remove before 1.0.

|L-4
|`HttpWellKnownResolver.getWellKnownResult()` `:131-133`
|**Test-only public method.** Zero production callers, only tests. Exposes the raw `WellKnownResult` object (which has mutable public fields), leaking internal details.
|Reduce to package-private or remove.

|L-5
|`StringSplitterMapper`
|**Zero production callers.** Not used by any `ClaimName` or any production code. Only referenced in `OAuthSheriffProcessor` for GraalVM reflection registration and in a test.
|Document as explicit public API for custom claim configurations, or remove. Can always be re-added.

|L-6
|`JwksType.NONE` `:32`
|**Sentinel enum value — only used in Quarkus health check.** No core production code uses it. Could be replaced by `Optional<JwksType>` or null.
|Consider removing; use null/Optional for "no JWKS configured."

|L-7
|`MapRepresentation` null checks on `data` `:115-127`
|**Unnecessary null guards.** Every method checks `if (data == null)`, but factory methods (lines 86-106) always return `Map.of()` for null/empty input. The only way to get null data is via the direct constructor `new MapRepresentation(null)`.
|Add null-coalescing in the compact constructor (`data = data != null ? data : Map.of()`) and remove per-method null checks.

|L-8
|`OAuthSheriffDevUIRuntimeService:209`
|**Dead null check for CDI-injected field.** `if (tokenValidator == null)` can never be true — `tokenValidator` is injected via `@Inject` constructor (line 64). CDI guarantees non-null constructor injection.
|Remove the dead null check.

|L-9
|`OAuthSheriffDevUIRuntimeService:78-80`
|**Test convenience constructor on production class.** Package-private two-arg constructor creates a default `ParserConfig`, disconnecting DevUI from runtime config in tests.
|Move to a test helper or provide all dependencies.

|L-10
|`DpopConfig` builder
|**No upper bound on `proofMaxAgeSeconds`.** Accepts `Long.MAX_VALUE`, which effectively disables proof freshness checking. Same for `nonceCacheTtlSeconds`.
|Add a reasonable upper bound (e.g., 3600s) or log WARN for unreasonably large values.

|L-11
|`AccessLogFilterConfig:122-136`
|**PathMatcher recompilation on every getter call.** `getIncludePathMatchers()`/`getExcludePathMatchers()` create new `PathMatcher` objects per invocation. `CustomAccessLogFilter` caches them (constructor), but the public methods are a performance trap for other callers.
|Cache lazily in the config, or remove from public API.

|L-12
|`AccessLogFilterConfig:44-53`
|**No validation of status code ranges.** `minStatusCode=600` or `minStatusCode > maxStatusCode` silently produces no logging.
|Validate: `minStatusCode` in `[100, 599]`, `maxStatusCode >= minStatusCode`.

|L-13
|`COULD_NOT_ACCESS_REQUEST` in `BearerTokenResult:186-194` / `BearerTokenStatus:56`
|**Intentional API never produced internally.** The factory method and enum value exist as documented public API for library consumers (Javadoc line 176-178), but `BearerTokenProducer` never produces this status. May confuse consumers who expect all statuses to be reachable.
|Document which statuses are produced by the built-in producer vs. available for custom implementations.

|L-14
|`WellKnownResult:46-49`
|**Public mutable fields cached after validation.** `issuer` and `jwksUri` are `public` mutable fields (required by DSL-JSON `@CompiledJson`). After deserialization and validation, the object is cached in `HttpWellKnownResolver`. Practical risk is low since the cache reference is internal.
|Copy to an immutable record after deserialization, or add a comment documenting the DSL-JSON constraint.

=== Key Themes

1. **Input validation at boundaries** (M-6, M-7, M-8, L-10, L-12): Builders and constructors accept values that would cause incorrect behavior at runtime. Pre-1.0 is the ideal time to tighten these.

2. **Consistency between pipelines** (M-1): The `IdTokenValidationPipeline` lacks the defensive null-checks present in `AccessTokenValidationPipeline`. Same author, same pattern — likely an oversight during the last refactor.

3. **Defense-in-depth** (M-2, M-4): Security controls exist at one layer but could be reinforced at adjacent layers. The main validation path is correct, and these are not exploitable through normal usage.

4. **Speculative API surface** (L-3, L-4, L-5, L-13): Methods and types that exist "in case someone needs them" but have zero internal callers. Remove or document before 1.0 to minimize the maintenance contract.

5. **Internal visibility** (M-3, M-9): Classes documented as internal (`@apiNote ... not part of the public API`) but declared `public`. Pre-1.0 is the time to enforce intended visibility.
