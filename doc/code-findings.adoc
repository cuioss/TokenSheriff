= Code Findings: Compatibility vs. Correct Design
:toc: left
:toclevels: 3
:sectnums:

== Overview

Semantic analysis of all production code asking one question per artifact:
_"Does this exist because it is the right design, or because we are afraid to break something?"_

Maturity: pre-1.0 — Breaking changes are free. No deprecation, no legacy support, no transitional code, no backward compatibility shims. Recommend removing/fixing everything.

Date: 2026-04-03

Modules analyzed: `oauth-sheriff-core` (107 files), `oauth-sheriff-quarkus` (42 files), `oauth-sheriff-quarkus-deployment` (2 files).

== Critical (Security Impact)

No critical security findings. The codebase consistently fails closed: unknown issuers, missing claims, invalid signatures, and unrecognized token types all throw `TokenValidationException`. DPoP validation correctly rejects null request context when a DPoP proof is present (DpopProofValidator:344-350).

== High (API / Design)

[cols="1,3,4,3"]
|===
|ID |Location |Issue |Recommendation

|H-1
|`TokenType.java:55-57, 87-96`
|`fromTypeClaim()` and `getTypeClaimName()` (Lombok getter on `typeClaimName`) have zero production callers across all modules. Only called in tests. The `typeClaimName` field ("Bearer", "ID", "Refresh") appears to be a leftover from when tokens were resolved via the body "typ" claim rather than the header-based `TokenHeaderValidator.validateTokenType()`.
|Remove `fromTypeClaim()` and `typeClaimName` field. `TokenType` should only carry `mandatoryClaims`.

|H-2
|`DecodedJwt.java:98-101`
|`DecodedJwt.of()` static factory has zero production callers across all modules (only called in `DecodedJwtTest`). All construction uses the record constructor directly. Redundant API surface.
|Remove the `of()` static factory method. The record constructor with compact constructor validation is sufficient.

|H-3
|`ClaimName.java:196-198`
|`ClaimName.map(MapRepresentation)` has zero production callers across all modules. `TokenBuilder` uses `claimMapper.map(mapRepresentation, key)` directly through the `allMappers` map, never through `ClaimName.map()`.
|Remove `ClaimName.map()`. The `claimMapper` field getter is still needed for `TokenBuilder`'s `allMappers` initialization.

|H-4
|`AccessTokenCache.java:169`
|`AccessTokenCache.get()` calls `OffsetDateTime.now()` on each cache hit verification, defeating the `ValidationContext.currentTime` optimization designed to eliminate system-call contention under high concurrency. The cache is checked BEFORE `ValidationContext` is created in the pipeline.
|Create `OffsetDateTime.now()` once at the start of `AccessTokenValidationPipeline.validate()` and pass it to `cache.get()`, aligning with the existing `ValidationContext` pattern.

|H-5
|`IssuerConfigResolver.java:465-481`
|`createHttpJwksLoaderConfig()` re-reads `ISSUER_IDENTIFIER` from config (line 469-472) even though `configureIssuerIdentifier()` at line 213-223 already reads the same property. The issuer identifier is read and validated twice from the same config source. Additionally, for well-known discovery the issuer identifier may be auto-discovered, yet this method unconditionally throws if absent.
|Pass the already-resolved issuer identifier as a parameter to `createHttpJwksLoaderConfig()` instead of re-reading. Verify behavior for the well-known discovery path where issuer identifier is auto-discovered.

|H-6
|`OAuthSheriffDevUIRuntimeService.java:260-274`
|`isJwtEnabled()` delegates to `countEnabledIssuers()`, which simply returns `issuerConfigs.size()`. Two-method indirection with no added value since `IssuerConfigResolver` already filters to enabled issuers only.
|Replace both with `!issuerConfigs.isEmpty()` inline at call sites, or keep `isJwtEnabled()` and delete `countEnabledIssuers()`.
|===

== Medium (Dead Code / Unnecessary API Surface)

[cols="1,3,4,3"]
|===
|ID |Location |Issue |Recommendation

|M-1
|`NonValidatingJwtParser.java:262-289`
|Token empty-check (lines 264-275) and size-check (lines 277-289) duplicate validation already performed by `TokenStringValidator.validate()`, which runs before the parser in all three pipelines (documented in pipeline javadocs: "TokenStringValidator has already checked: null, blank, size"). Each size check allocates a `token.getBytes(UTF_8)` byte array.
|Remove the empty-check and size-check from `decodeInternal()`. Document the `TokenStringValidator` precondition. If defense-in-depth is desired, use an assert instead.

|M-2
|`CollectionClaimHandler.java:71-76`
|`getValues()` is public but only called internally by `providesValues()` and `determineMissingValues()` within the same class. Never called externally in production code (only tests).
|Change visibility to private. External callers should use `providesValues()` or `determineMissingValues()`.

|M-3
|`SignatureAlgorithmPreferences.java:80-85`, `JwkAlgorithmPreferences.java:68-74`
|`getDefaultPreferredAlgorithms()` and `getDefaultSupportedAlgorithms()` are `public static` methods but only called from their respective no-arg constructors (and tests). Exposes the exact default algorithm list as API surface.
|Make both methods private. Callers needing the default list can construct the no-arg instance.

|M-4
|`TokenClaimValidator.java:96-108`
|Constructor emits WARN-level log for missing `expectedTokenType` (line 106) even though it's explicitly opt-in by design. This fires for virtually every real-world deployment since most IdPs don't support RFC 9068 `typ` claims. Creates log noise.
|Downgrade `expectedTokenType` warning to DEBUG. For `expectedAudience` warning (line 97), check `audienceValidationDisabled` before emitting.

|M-5
|`IssuerConfig.java:1001-1013`
|Private constructor silently replaces null with defaults for `expectedAudience` (→ `Set.of()`), `expectedClientId` (→ `Set.of()`), `algorithmPreferences` (→ new instance), and `claimMappers` (→ `Map.of()`). While the builder already initializes these defaults, the null-coalescing masks potential builder bugs.
|Use `Objects.requireNonNull` in the constructor instead of null-coalescing, since the builder already provides defaults. This catches builder bugs early.

|M-6
|`BearerTokenResponseFactory.java:86-93`
|`createResponse()` handles `FULLY_VERIFIED` → `createSuccessResponse()`, but the only production caller is `BearerTokenResult.createErrorResponse()` which throws `IllegalStateException` for `FULLY_VERIFIED` status first. The `FULLY_VERIFIED` branch is unreachable via the current call path.
|Either make `createResponse()` package-private (since only `BearerTokenResult.createErrorResponse` calls it), or document that FULLY_VERIFIED handling exists for completeness of the public static method.

|M-7
|`JwtMetricsCollector.java:191-203`
|`clear()` is public but only called from tests. No production code invokes it.
|Mark as test-only via comment, or make package-private. If intended as operational API (management endpoint), document that intent.

|M-8
|`TokenValidatorMonitor.java:174-177, 184-200`
|`getSampleCount()` (public) and `reset()`/`resetAll()` (package-private) have zero production callers in any module. Only called in tests.
|Make `getSampleCount()` package-private or remove. Keep `reset()`/`resetAll()` only if test utility is needed.
|===

== Low (Cleanup)

[cols="1,3,4,3"]
|===
|ID |Location |Issue |Recommendation

|L-1
|`NonValidatingJwtParser.java:262`
|`decodeInternal(String, boolean, boolean)` uses two boolean parameters (`logWarnings`, `trackSecurityEvents`), making call sites opaque: `decodeInternal(token, true, true)`. Method is private with 3 callers.
|Consider a `DecodeMode` enum (`NORMAL`, `QUIET`, `OPAQUE`) for readability. Low priority since the method is private.

|L-2
|`IssuerConfigCache.java:88`
|Package-private constructor does not validate that `issuerConfigs` and `securityEventCounter` are non-null. Null would throw `NullPointerException` at line 98 with no descriptive message.
|Add `Objects.requireNonNull` with descriptive messages for both parameters.

|L-3
|`CachedToken.java:71-73`
|`isExpired()` uses `currentTime.minusSeconds(clockSkewSeconds)` while `ValidationContext.isExpired()` uses `expirationTime.plusSeconds(clockSkewSeconds)`. Both are mathematically equivalent but the inconsistent expression style could confuse maintainers.
|Align with `ValidationContext` convention: `expirationTime.plusSeconds(clockSkewSeconds).isBefore(currentTime)`.

|L-4
|`MinimalTokenContent.java:42` → `BaseTokenContent.java:54`
|`MinimalTokenContent` extends `Serializable`, inherited by the entire token hierarchy. The sealed hierarchy + records pattern with in-memory validation pipeline and `ConcurrentHashMap` cache suggests serialization is never actually used in production.
|Verify if `Serializable` is needed. If no serialization occurs (no session replication, no distributed cache), remove from `MinimalTokenContent`.

|L-5
|`CustomAccessLogFilter.java:69`
|`@Nullable` annotation on CDI `Instance<HttpServerRequest> vertxRequest` is misleading — CDI `Instance` injection points are never null. The null check at line 207 is unnecessary.
|Remove `@Nullable` and the null check. Keep the `isUnsatisfied()` check which is the correct CDI `Instance` guard.

|L-6
|`OAuthSheriffDevUIRuntimeService.java:148-153`
|DevUI displays hardcoded strings `"connectTimeoutSeconds", "per-issuer (default: 10)"`. If actual defaults in `HttpJwksLoaderConfig` change, these become stale.
|Read actual defaults from issuer configs (already available), or add a comment noting these are approximate display values.

|L-7
|`BearerTokenResult.java:72`
|Implements `Serializable` with `@SuppressWarnings("java:S1948")` for a request-scoped CDI bean. Serialization is not typically needed for request-scoped beans.
|Remove `Serializable` unless there's a concrete use case (session replication, distributed cache). If one exists, document it.

|L-8
|`IssuerConfigResolver.java:465`
|`createHttpJwksLoaderConfig` parameters are `@Nullable String jwksUrl, @Nullable String wellKnownUrl`, but the caller at line 435 only invokes it when at least one is present. The `@Nullable` annotations are technically correct but misleading about actual usage.
|Add a precondition: `Preconditions.checkArgument(jwksUrl != null \|\| wellKnownUrl != null)` to document the invariant.

=== Dismissed False Positives

The following candidates were investigated and dismissed during Phase 5 cross-module verification:

* `SecurityEventCounter.getCount()` / `getCounters()` / `reset()` — reported as unused in `core`, but all three are called from `JwtMetricsCollector` in the `quarkus` module (production code).
* `AccessTokenContent` `*AndDebugIf*Missing` methods — has explicit Design Decision comment confirming intentional public API for library consumers with zero internal callers by design.
* `BearerTokenProducer.resolveRequestContext()` silent DPoP fallback — core `DpopProofValidator:344-350` throws `TokenValidationException` when request URI/method is null and a DPoP proof is present; core fails closed regardless of what the quarkus module passes.
