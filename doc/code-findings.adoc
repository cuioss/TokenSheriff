= Code Findings: Compatibility vs. Correct Design
:toc: left
:toclevels: 3
:sectnums:

== Overview

Semantic analysis of all production code asking one question per artifact:
_"Does this exist because it is the right design, or because we are afraid to break something?"_

Maturity: pre-1.0 — Breaking changes are free. No deprecation, no legacy support, no transitional code. Recommend removing/fixing everything.

Date: 2026-04-03

Status: **All findings resolved** (12 fixed, 1 kept as-is per decision)

Scope: `oauth-sheriff-core`, `oauth-sheriff-quarkus`, `oauth-sheriff-quarkus-deployment`

== Critical (Security Impact)

No critical findings. All security-relevant defaults are properly documented and default to the secure option.

== High (API / Design)

[cols="1,3,4,3"]
|===
|ID |Location |Issue |Recommendation

|H-1
|`AudienceValidator.java:98-99, 110-118`
|When `accessTokenAudienceOptional=true` and the audience claim is missing from an access token, the skip is only logged at DEBUG level (line 99). Similarly, the azp-as-audience fallback (lines 110-118) silently substitutes the `azp` claim for the missing `aud` claim, also at DEBUG level only. Both represent security-relevant behavior that operators should be able to detect without enabling DEBUG logging.
|Log the `accessTokenAudienceOptional` skip at INFO level. Log the azp-as-audience fallback at INFO level. These are not errors, but operators need visibility into when their IdP is not providing expected claims.

|H-2
|`AuthorizedPartyValidator.java:73-80`
|When `expectedClientId` is configured, the validator requires the `azp` claim to be present and rejects tokens without it. However, `azp` is OPTIONAL per OIDC Core 1.0, and RFC 9068 Section 2.2 specifies `client_id` as the standard claim for JWT access tokens. Tokens from IdPs that emit `client_id` but not `azp` will be incorrectly rejected.
|Add a fallback to check the `client_id` claim (RFC 9068) when `azp` is absent. This broadens compatibility without weakening security, since both claims serve the same purpose.

|H-3
|`BearerTokenResponseFactory.java:88`
|The `FULLY_VERIFIED` case in `createResponse()` (line 88) calls `createSuccessResponse()`, but the only caller of `createResponse()` is `BearerTokenResult.createErrorResponse()` (line 242 of `BearerTokenResult.java`), which throws `IllegalStateException` for `FULLY_VERIFIED` status (lines 239-241). The `FULLY_VERIFIED` branch and `createSuccessResponse()` method are dead code that can never be reached.
|Remove the `FULLY_VERIFIED` case from the switch and the `createSuccessResponse()` method. Replace with a `default` case that throws `IllegalStateException` to fail fast if a new status is added without updating the factory.

|===

== Medium (Dead Code / Unnecessary API Surface)

[cols="1,3,4,3"]
|===
|ID |Location |Issue |Recommendation

|M-1
|`MeasurementType.java:82`
|`TOKEN_FORMAT_CHECK` enum constant exists but is never used for instrumentation in production code. `TokenStringValidator.java:42-43` mentions it as a "Future Enhancement" in its Javadoc but never creates a ticker for it. The constant is only referenced in test code.
|Either instrument `TokenStringValidator` with this measurement type or remove the unused enum constant. Pre-1.0 is the right time to avoid speculative API surface.

|M-2
|`MetricsTickerFactory.java:45-51`
|`createTicker()` (the non-started variant) is public but has zero production callers. It is only called internally by `createStartedTicker()` (line 71) and by test code. The two-step "create then start" pattern is never used in production.
|Make `createTicker()` private or remove it.

|M-3
|`UnvalidatedRefreshToken.java:65, 82-84`
|The `claims` field is populated by `RefreshTokenValidationPipeline` (line 105 of `RefreshTokenValidationPipeline.java`) and stored for consumer access. However, the `getClaims()` accessor is package-private (line 82), making the claims inaccessible to library consumers. `UnvalidatedRefreshToken` implements `MinimalTokenContent` (not `TokenContent`), so there is no public claims API. Consumers who receive JWT-formatted refresh tokens (e.g., to inspect `exp` for proactive renewal) have no way to access the claims that are already parsed and stored.
|Make `getClaims()` public to match the design intent (claims are deliberately parsed and stored). The existing `@WARNING` Javadoc on the class already clearly communicates that these claims are unvalidated and must not be used for authorization decisions.

|M-4
|`JwtMetricsCollector.java:107`
|`JwtMetricsCollector` injects `TokenValidator` and extracts `SecurityEventCounter` via `tokenValidator.getSecurityEventCounter()` in `@PostConstruct`. However, `TokenValidatorProducer` already produces `SecurityEventCounter` as a `@Produces @ApplicationScoped` CDI bean (lines 85-87 of `TokenValidatorProducer.java`). The metrics collector could inject `SecurityEventCounter` directly, eliminating the unnecessary coupling to `TokenValidator`.
|Inject `SecurityEventCounter` directly via CDI instead of extracting it from `TokenValidator`. Remove the `TokenValidator` dependency from `JwtMetricsCollector`.

|M-5
|`JsonWebTokenAdapter.java:142-144`
|`getTokenContent()` is package-private and only called from `JsonWebTokenAdapterTest`. No production code accesses the underlying `TokenContent` delegate through this method.
|Remove the method. Tests can verify behavior through the `JsonWebToken` interface methods.

|===

== Low (Cleanup)

[cols="1,3,4,3"]
|===
|ID |Location |Issue |Recommendation

|L-1
|`DecodedJwt.java:235-238`
|`redactToken()` returns tokens of 10 characters or fewer unredacted. While valid JWTs are always much longer than 10 characters, the function name implies redaction regardless of length. If this method is ever used for non-JWT tokens, short tokens would leak.
|Always redact (e.g., show first 4 characters for short tokens), or rename to clarify the length-dependent behavior.

|L-2
|`BearerTokenProducer.java:229-239`
|`resolveRequestContext()` returns a 2-element `String[]` where `[0]` is the URI and `[1]` is the HTTP method. The array elements are accessed by magic indices throughout the class, making the code fragile and semantically opaque.
|Replace with a simple record: `record RequestContext(String uri, String method) {}`.

|L-3
|`OAuthSheriffDevUIRuntimeService.java:143-153`
|Hardcoded default values like `"per-issuer (default: 10)"` and a fallback `60` for clock skew. The comment acknowledges `HttpJwksLoaderConfig` is not accessible, but these values will silently diverge if core defaults change.
|Reference actual values from injected `IssuerConfig` objects or constants from the core module where possible.

|L-4
|`AccessTokenCacheConfigResolver.java:58-59`
|Hardcodes `maxSize=1000` and `evictionIntervalSeconds=10L`. If `AccessTokenCacheConfig` builder defaults diverge, these will silently conflict. Other resolvers (e.g., `ParserConfigResolver`) use the builder-defaults-first pattern correctly.
|Only call setters when properties are explicitly configured, letting the builder provide defaults. Follow the pattern used by `ParserConfigResolver`.

|L-5
|`BearerTokenResult.java:72`
|`BearerTokenResult` implements `Serializable` with `serialVersionUID`, but it is a transient per-request value object. Pre-1.0 is the right time to question whether serialization support is needed (HTTP session replication, cluster failover scenarios).
|Evaluate whether `Serializable` is truly needed. If not, remove it and the `@SuppressWarnings` annotation.

=== Dismissed False Positives

The following candidates were identified during per-module analysis but dismissed during cross-module verification:

* `TokenContent.getClaimNames()` reported as "zero production callers" -- dismissed because `JsonWebTokenAdapter.getClaimNames()` in the `oauth-sheriff-quarkus` module delegates to this method.
* `SecurityEventCounter.reset()` reported as "test-only" -- dismissed because `JwtMetricsCollector.java:196` in the `oauth-sheriff-quarkus` module calls `reset()` in production code.
* `MetricIdentifier.VALIDATION.*` constants reported as having no external callers -- dismissed because public metric name constants are a reasonable API for consumers querying Micrometer metrics programmatically.
