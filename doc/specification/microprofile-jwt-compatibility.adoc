== Overview

This document describes OAuthSheriff's relationship to the https://download.eclipse.org/microprofile/microprofile-jwt-auth-2.1/microprofile-jwt-auth-spec-2.1.html[Eclipse MicroProfile JWT Auth 2.1] specification. OAuthSheriff implements select MP-JWT 2.1 primitives as a compatibility layer while retaining its advanced multi-issuer, multi-token-type architecture.

== Design Rationale

OAuthSheriff is **not** an MP-JWT implementation. It is a purpose-built, production-grade JWT/OAuth 2.0/OpenID Connect token validation library that provides capabilities beyond what MP-JWT defines. Rather than conforming to the MP-JWT specification's single-issuer, single-token-type model, OAuthSheriff adopts **select primitives** from the spec where they provide value.

The approach is:

* **Core types**: `TokenContent` is framework-agnostic â€” no dependency on `JsonWebToken`
* **Quarkus adapter**: `JsonWebTokenAdapter` wraps `TokenContent` to provide the `JsonWebToken` interface for CDI injection
* **Clean API separation**: Core uses `Optional`-returning accessors; the adapter bridges to MP-JWT's nullable contract
* **OAuthSheriff extensions**: Multi-issuer support, DPoP, custom claim mappers, sealed token hierarchy, security event tracking â€” all preserved as additional API surface

== Adopted Features

=== `JsonWebToken` Interface (`org.eclipse.microprofile.jwt.JsonWebToken`)

The Quarkus module provides a `JsonWebTokenAdapter` that wraps `TokenContent` and implements `JsonWebToken`, making it available as a `java.security.Principal`. This enables standard Jakarta Security integration:

[source,java]
----
@Inject JsonWebToken callerPrincipal;  // MP-JWT standard injection via adapter
@Inject Principal principal;            // Jakarta Security standard injection
----

The adapter bridges between OAuthSheriff's clean API and the MP-JWT spec signatures:

[cols="3,2,2"]
|===
|JsonWebToken Method |Core TokenContent API |Notes

|`getName()`
|`getClaimOption(UPN)` / `getSubject()`
|UPN fallback: `upn` -> `preferred_username` -> `sub`

|`getSubject()`
|`getSubject()` -> `Optional<String>`
|Adapter returns nullable; core returns Optional

|`getExpirationTime()`
|`getExpirationDateTime()` -> `OffsetDateTime`
|Adapter returns epoch seconds; core returns rich type

|`getIssuedAtTime()`
|`getIssuedAtDateTime()` -> `OffsetDateTime`
|Adapter returns epoch seconds; core returns rich type

|`getGroups()`
|`AccessTokenContent.getGroups()`
|Only on concrete type in core

|`getAudience()`
|`AccessTokenContent.getAudience()` / `IdTokenContent.getAudience()`
|Only on concrete types in core

|`getClaimNames()`
|`getClaimNames()`
|Same signature on both

|`getClaim(String)`
|`getClaimOption(ClaimName)`
|Adapter provides typed conversion
|===

=== `upn` Claim (User Principal Name)

The `upn` claim is MP-JWT's standardized human-readable principal identifier. OAuthSheriff's `ClaimName` enum includes `UPN` with the spec-defined fallback chain for `Principal.getName()`:

1. `upn` claim (if present)
2. `preferred_username` claim (fallback)
3. `sub` claim (final fallback)

=== Clock Skew Tolerance (`mp.jwt.verify.clock.skew`)

Clock skew tolerance is applied to both `exp` and `nbf` validation, accommodating clock drift in distributed systems. A token is considered expired only when `exp + clockSkew < now`.

Configured per issuer via `sheriff.oauth.issuers.<name>.clock-skew-seconds` (default: 60 seconds).

=== Token Age Verification (`mp.jwt.verify.token.age`)

Maximum token age validation based on `iat` claim. When enabled, rejects tokens where `now - iat > maxTokenAgeSeconds + clockSkewSeconds`, even if not yet expired.

Configured per issuer via `sheriff.oauth.issuers.<name>.max-token-age-seconds` (default: disabled).

=== CDI Producers (`@Produces JsonWebToken`, `@Produces Principal`)

Standard MP-JWT CDI injection is supported:

[source,java]
----
@Inject JsonWebToken callerPrincipal;  // MP-JWT standard
@Inject Principal principal;            // Jakarta Security standard
----

When no valid token is present, an empty `JsonWebToken` is injected (per MP-JWT spec).

=== Cookie-Based Token Extraction (`mp.jwt.token.header` / `mp.jwt.token.cookie`)

Alternative token extraction from HTTP cookies instead of the `Authorization` header. Useful for browser-based SPAs using SameSite cookie protection.

Configuration:

* `sheriff.oauth.token.header` â€” `Authorization` (default) or `Cookie`
* `sheriff.oauth.token.cookie-name` â€” cookie name (default: `Bearer`)

When `sheriff.oauth.token.header=Cookie`, the token is extracted from the named cookie in the `Cookie` HTTP header. If a valid `Authorization: Bearer <token>` header is also present, it takes precedence.

== Not Adopted (By Design)

[cols="2,3"]
|===
|MP-JWT Feature |Rationale

|`@RolesAllowed` annotation
|OAuthSheriff uses `@BearerAuth` which provides superior multi-dimensional authorization (scopes AND roles AND groups). The standard `@RolesAllowed` only checks one dimension.

|`@Claim` CDI qualifier injection
|OAuthSheriff's programmatic `getClaimOption(ClaimName)` with custom `ClaimMapper` pipeline provides type-safe, extensible claim access. `@Claim` injection is less flexible for custom claim types and multi-issuer scenarios.

|`@LoginConfig` annotation
|Not applicable to Quarkus extension model. OAuthSheriff activates via `@BearerAuth` interceptor binding and configuration properties.

|`mp.jwt.*` config namespace
|`sheriff.oauth.*` is the established convention. OAuthSheriff's per-issuer configuration model (`sheriff.oauth.issuers.<name>.*`) has no equivalent in the single-issuer `mp.jwt.*` namespace.

|`ClaimValue<T>` (MP-JWT version)
|OAuthSheriff's `ClaimValue` with `ClaimMapper` pipeline (e.g., `ScopeMapper`, `KeycloakDefaultRolesMapper`) is more powerful. MP-JWT's `ClaimValue` is a simple typed wrapper.

|Single-issuer model
|OAuthSheriff's multi-issuer `IssuerConfigCache` with per-issuer JWKS loading, algorithm constraints, and claim validation is a core differentiator. MP-JWT assumes one issuer per application.
|===

== OAuthSheriff Advantages Beyond MP-JWT

The following capabilities have no equivalent in MP-JWT 2.1:

* **Multi-issuer support** â€” `IssuerConfigCache` with async JWKS loading per issuer
* **DPoP (RFC 9449)** â€” Optional Demonstrating Proof-of-Possession validation
* **Custom claim mappers** â€” Pluggable `ClaimMapper` interface with Keycloak-specific implementations
* **Custom validation rules** â€” Pluggable `TokenValidationRule` interface for domain-specific post-validation checks (e.g., tenant-id enforcement, subject blocklists)
* **Sealed token hierarchy** â€” `AccessTokenContent`, `IdTokenContent`, `RefreshTokenContent` with type-safe access patterns
* **Security event tracking** â€” `SecurityEventCounter` with categorized validation failure tracking
* **Performance monitoring** â€” `TokenValidatorMonitor` with microsecond-precision pipeline measurements
* **Token caching** â€” `AccessTokenCache` with TTL-based eviction
* **JWE transparency** â€” Encrypted tokens decrypted transparently before validation
* **Algorithm security** â€” RFC 8725 and NIST SP 800-131A compliant algorithm whitelisting
