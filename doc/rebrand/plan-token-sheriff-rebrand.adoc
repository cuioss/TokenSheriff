== Goal

Rename the project from *OAuth-Sheriff* to *Token-Sheriff* across every surface â€” repository, Maven coordinates, Java packages, configuration keys, metrics, documentation, and the requirement-ID scheme â€” and provide a clean migration path for the one consumer (`nifi-extensions`).

The name "OAuth" is inaccurate: this is a multi-issuer *JWT validation* library that already does OIDC Discovery and ID-token validation, and will grow an xref:../oidc/decision-oidc-module.adoc[OIDC relying-party module]. "Token-Sheriff" reflects the enduring scope (tokens, with headroom for the RP work).

== Name verification (done 2026-06-24)

The name is clear to take:

* *Maven Central:* `search.maven.org` returns *0* artifacts for `token-sheriff`. The `de.cuioss` groupId namespace is ours, so any `de.cuioss.*` coordinate is guaranteed free.
* *GitHub / npm:* no library or package named `token-sheriff` / `TokenSheriff`.
* *Adjacency (acceptable, documented):* the bare "Sheriff" is crowded in security tooling (sheriff-software.com software protection; `elementsinteractive/sheriff` repo scanner). "Crypto Sheriff" is Europol/No-More-Ransom's ransomware tool, and "Sheriff Inu" is an ERC-20 token. None collide with the qualified, hyphenated "Token-Sheriff"; the closest semantic neighbour is "Crypto Sheriff" (a different domain).

Sources: https://search.maven.org/[Maven Central Search], https://maven.apache.org/guides/mini/guide-relocation.html[Maven relocation guide], https://www.nomoreransom.org/crypto-sheriff.php[No More Ransom â€” Crypto Sheriff].

== Adjacent naming schemes â€” explicit decisions

* *Requirement IDs `OAUTH-SHERIFF-N`* â†’ rename to `TOKEN-SHERIFF-N`. Mechanical global replace, but it spans `doc/Requirements.adoc` anchors and every `@see` / spec reference in code and tests â€” must be done atomically with the rest so traceability stays intact.

* *Metrics `sheriff.oauth.*`* (`MetricIdentifier.PREFIX`) â†’ `sheriff.token.*`. User-facing in dashboards/Prometheus â€” call out in release notes.
* *Class / feature names* (`OAuthSheriffProcessor`, the Quarkus feature string, `OAUTH_SHERIFF_*` log constants) â†’ Token-Sheriff equivalents.
* *Mascot* `doc/resources/Sheriff.png` and the "Sheriff" brand â†’ unchanged (the brand survives; only the qualifier changes).
* *No `module-info.java`* present â†’ no JPMS module names to migrate.

== Tooling: OpenRewrite for the structural layer

The Java type/package and Maven-coordinate refactoring **must** be done with OpenRewrite, not text substitution. `de.cuioss.sheriff.oauth` is a substring of unrelated things (the word "oauth", jar names, prose); `sed` cannot tell a package reference from a string literal or an identifier, and it leaves source files in their old directories. OpenRewrite is AST-aware, relocates the files to match the new package, is idempotent, and produces a reviewable diff. This repository already runs OpenRewrite (`cui-rewrite`), so it fits the existing toolchain.

Split by what is structural (AST) vs textual:

[cols="2,3"]
|===
| Surface | Tool

| Java packages, imports, FQNs, static imports, Javadoc `@link`, file relocation
| `org.openrewrite.java.ChangePackage` (`de.cuioss.sheriff.oauth` â†’ `de.cuioss.sheriff.token`, recursive)

| Specific renamed types (`OAuthSheriffProcessor` â†’ `TokenSheriffProcessor`, log-constant holders)
| `org.openrewrite.java.ChangeType`

| Maven dependency + managed-dependency coordinates (in this repo and consumers)
| `org.openrewrite.maven.ChangeDependencyGroupIdAndArtifactId`, `ChangeManagedDependencyGroupIdAndArtifactId`

| Parent POM coordinates
| `org.openrewrite.maven.ChangeParentPom`

| Config / metric key *string literals* and `.properties` keys (`sheriff.oauth` â†’ `sheriff.token`)
| `org.openrewrite.properties.ChangePropertyKey` + `org.openrewrite.text.FindAndReplace` (Java literals, regex-scoped)

| Requirement IDs `OAUTH-SHERIFF-N`, docs, badges, workflows, README `alt`
| `org.openrewrite.text.FindAndReplace` (or scripts) â€” not an AST concern
|===

[NOTE]
====
No first-class recipe changes a module's *own* `<groupId>`/`<artifactId>`. Those ~6 poms (`token-sheriff-parent`, `-core`, `-quarkus*`, `bom`) are changed manually / via IDE or a scoped `text.FindAndReplace` â€” small and low-risk. The bulk (the ~1955 package references and all consumer dependency coordinates) is the part OpenRewrite makes safe.
====

Compose the recipes into a single `rewrite.yml` and run `./mvnw rewrite:run` (or the configured `-Ppre-commit` recipe pass), so the structural rename is one reviewable, idempotent step.

*Ship a consumer migration recipe.* Publish the same composite as a named recipe (e.g. `de.cuioss.sheriff.token.MigrateFromOAuthSheriff`) so consumers â€” starting with `nifi-extensions` â€” run `mvn rewrite:run` to auto-migrate coordinates, imports, and config keys instead of hand-editing. See the xref:migration-nifi-extensions.adoc[migration guide].

== Maven Central relocation (answering "can Central indicate the move?")

*Yes.* Maven supports artifact *relocation*: publish a final POM under the OLD coordinates whose `distributionManagement/relocation` points to the NEW coordinates. Resolvers automatically redirect to the new artifact and print a warning telling consumers to update.

[source,xml]
----
<project xmlns="http://maven.apache.org/POM/4.0.0">
  <modelVersion>4.0.0</modelVersion>
  <!-- OLD coordinates -->
  <groupId>de.cuioss.sheriff.oauth</groupId>
  <artifactId>oauth-sheriff-core</artifactId>
  <version>1.0.0</version>
  <distributionManagement>
    <relocation>
      <groupId>de.cuioss.sheriff.token</groupId>
      <artifactId>token-sheriff-core</artifactId>
      <!-- version optional if unchanged -->
      <message>Renamed to Token-Sheriff. See migration guide.</message>
    </relocation>
  </distributionManagement>
</project>
----

Consumer experience:

----
[WARNING] The artifact de.cuioss.sheriff.oauth:oauth-sheriff-core:jar:1.0.0
          has been relocated to de.cuioss.sheriff.token:token-sheriff-core:jar:1.0.0
----

Notes:

* Publish a relocation stub for *each* old artifact (`-core`, `-quarkus`, `-quarkus-deployment`, `-parent`, `bom`).
* Relocation redirects *coordinates only*. It does **not** rename Java packages or config keys â€” consumers must still adopt those. The grace window is exactly what the xref:migration-nifi-extensions.adoc[migration guide] uses.
* Source: https://maven.apache.org/guides/mini/guide-relocation.html[Apache Maven â€” Guide to relocation].

== Risks & mitigations

* *Atomicity* â€” packages, config keys, and requirement IDs must all flip together; a partial rename is worse than none. Mitigation: one branch, scripted transforms, full build gate before merge.
* *Hidden string refs* â€” config/metric keys live in `.properties`, `.adoc`, and test literals, not just Java. Mitigation: the inventory greps above are the regression baseline; re-run them post-rename expecting ~0 for old tokens.
* *External-service lag* â€” Sonar/Pages/OpenSSF update out of band. Mitigation: Phase 3 checklist; verify each badge.
* *Consumer breakage* â€” `nifi-extensions` build breaks the moment it bumps to the new coordinates. Mitigation: relocation grace window + the migration guide + coordinating the bump.
