== Goal

Rename the project from *OAuth-Sheriff* to *Token-Sheriff* across every surface — repository, Maven coordinates, Java packages, configuration keys, metrics, documentation, and the requirement-ID scheme — and provide a clean migration path for the one consumer (`nifi-extensions`).

The name "OAuth" is inaccurate: this is a multi-issuer *JWT validation* library that already does OIDC Discovery and ID-token validation, and will grow an xref:../oidc/decision-oidc-module.adoc[OIDC relying-party module]. "Token-Sheriff" reflects the enduring scope (tokens, with headroom for the RP work).

== Name verification (done 2026-06-24)

The name is clear to take:

* *Maven Central:* `search.maven.org` returns *0* artifacts for `token-sheriff`. The `de.cuioss` groupId namespace is ours, so any `de.cuioss.*` coordinate is guaranteed free.
* *GitHub / npm:* no library or package named `token-sheriff` / `TokenSheriff`.
* *Adjacency (acceptable, documented):* the bare "Sheriff" is crowded in security tooling (sheriff-software.com software protection; `elementsinteractive/sheriff` repo scanner). "Crypto Sheriff" is Europol/No-More-Ransom's ransomware tool, and "Sheriff Inu" is an ERC-20 token. None collide with the qualified, hyphenated "Token-Sheriff"; the closest semantic neighbour is "Crypto Sheriff" (a different domain).

Sources: https://search.maven.org/[Maven Central Search], https://maven.apache.org/guides/mini/guide-relocation.html[Maven relocation guide], https://www.nomoreransom.org/crypto-sheriff.php[No More Ransom — Crypto Sheriff].

[IMPORTANT]
.One decision to ratify before execution
====
The current convention is *reversed* relative to the brand: groupId/config are `sheriff.oauth` while the brand is "OAuth-Sheriff". This plan keeps that convention (`de.cuioss.sheriff.token`, `sheriff.token.*`) because it changes only the `oauth → token` leaf and keeps the stable `de.cuioss.sheriff.*` / `sheriff.*` roots — minimising consumer churn.

Alternative (brand-order alignment, larger change): `de.cuioss.token.sheriff` + `token.sheriff.*`, or `de.cuioss.tokensheriff`. Decide before Phase 1; everything downstream depends on it.
====

== Adjacent naming schemes — explicit decisions

* *Requirement IDs `OAUTH-SHERIFF-N`* → rename to `TOKEN-SHERIFF-N`. Mechanical global replace, but it spans `doc/Requirements.adoc` anchors and every `@see` / spec reference in code and tests — must be done atomically with the rest so traceability stays intact.
* *Config keys `sheriff.oauth.*`* (`JwtPropertyKeys.PREFIX`) → `sheriff.token.*`. User-facing/breaking — covered by relocation grace window + migration guide.
* *Metrics `sheriff.oauth.*`* (`MetricIdentifier.PREFIX`) → `sheriff.token.*`. User-facing in dashboards/Prometheus — call out in release notes.
* *Class / feature names* (`OAuthSheriffProcessor`, the Quarkus feature string, `OAUTH_SHERIFF_*` log constants) → Token-Sheriff equivalents.
* *Mascot* `doc/resources/Sheriff.png` and the "Sheriff" brand → unchanged (the brand survives; only the qualifier changes).
* *No `module-info.java`* present → no JPMS module names to migrate.

== Execution plan

=== Phase 0 — Decisions & prerequisites
* Ratify the groupId/config scheme (see decision box above).
* Confirm the cut release: do the rebrand as part of the first post-rename version (proposed `1.0.0`).
* Snapshot the current published coordinates/versions (needed for relocation stubs).

=== Phase 1 — In-repo rename (single atomic branch)
Order matters; do it as scripted, reviewable transforms:

. Rename module directories `oauth-sheriff-* → token-sheriff-*`.
. Move Java packages `de.cuioss.sheriff.oauth → de.cuioss.sheriff.token` (source tree move + package declarations + imports).
. Update all POMs: `groupId`, `artifactId`, parent refs, BOM, `<name>`/`<url>`.
. Flip prefix constants `JwtPropertyKeys.PREFIX` and `MetricIdentifier.PREFIX` to `sheriff.token`; update every `sheriff.oauth.*` literal in `application*.properties`, tests, and docs.
. Rename requirement IDs `OAUTH-SHERIFF-N → TOKEN-SHERIFF-N` (Requirements.adoc + all references).
. Rename classes/features (`OAuthSheriffProcessor`, Quarkus feature, log constants).
. Update all docs, badges, README image `alt`, `quarkus-extension.yaml`, GitHub workflows, `.github/project.yml` Sonar key, and `cuioss.github.io/OAuth-Sheriff` URLs.
. Build green: `./mvnw -Ppre-commit clean verify` then `./mvnw clean install`.

=== Phase 2 — Maven Central relocation
Publish a final *relocation release* under the old coordinates so existing builds auto-redirect (details below).

=== Phase 3 — GitHub repo + external services
* Rename the GitHub repo `OAuthSheriff → TokenSheriff` (GitHub keeps redirects for the old URL/clone paths).
* Update SonarCloud project (new key `cuioss_TokenSheriff`), GitHub Pages (verify the new base URL — note the current Pages path is hyphenated `OAuth-Sheriff` while the repo is `OAuthSheriff`; reconcile this during the move), and the OpenSSF Best Practices entry.
* Verify all badges resolve.

=== Phase 4 — Consumer migration
Execute xref:migration-nifi-extensions.adoc[the nifi-extensions migration] against the first Token-Sheriff release.

=== Phase 5 — Verification
See the checklist below.

== Maven Central relocation (answering "can Central indicate the move?")

*Yes.* Maven supports artifact *relocation*: publish a final POM under the OLD coordinates whose `distributionManagement/relocation` points to the NEW coordinates. Resolvers automatically redirect to the new artifact and print a warning telling consumers to update.

[source,xml]
----
<project xmlns="http://maven.apache.org/POM/4.0.0">
  <modelVersion>4.0.0</modelVersion>
  <!-- OLD coordinates -->
  <groupId>de.cuioss.sheriff.oauth</groupId>
  <artifactId>oauth-sheriff-core</artifactId>
  <version>1.0.0</version>
  <distributionManagement>
    <relocation>
      <groupId>de.cuioss.sheriff.token</groupId>
      <artifactId>token-sheriff-core</artifactId>
      <!-- version optional if unchanged -->
      <message>Renamed to Token-Sheriff. See migration guide.</message>
    </relocation>
  </distributionManagement>
</project>
----

Consumer experience:

----
[WARNING] The artifact de.cuioss.sheriff.oauth:oauth-sheriff-core:jar:1.0.0
          has been relocated to de.cuioss.sheriff.token:token-sheriff-core:jar:1.0.0
----

Notes:

* Publish a relocation stub for *each* old artifact (`-core`, `-quarkus`, `-quarkus-deployment`, `-parent`, `bom`).
* Relocation redirects *coordinates only*. It does **not** rename Java packages or config keys — consumers must still adopt those. The grace window is exactly what the xref:migration-nifi-extensions.adoc[migration guide] uses.
* Source: https://maven.apache.org/guides/mini/guide-relocation.html[Apache Maven — Guide to relocation].

== Risks & mitigations

* *Atomicity* — packages, config keys, and requirement IDs must all flip together; a partial rename is worse than none. Mitigation: one branch, scripted transforms, full build gate before merge.
* *Hidden string refs* — config/metric keys live in `.properties`, `.adoc`, and test literals, not just Java. Mitigation: the inventory greps above are the regression baseline; re-run them post-rename expecting ~0 for old tokens.
* *External-service lag* — Sonar/Pages/OpenSSF update out of band. Mitigation: Phase 3 checklist; verify each badge.
* *Consumer breakage* — `nifi-extensions` build breaks the moment it bumps to the new coordinates. Mitigation: relocation grace window + the migration guide + coordinating the bump.
