== Goal

Produce, _before_ any RP code, the document set that mirrors the existing validation docs (`Requirements.adoc`, `specification/`, `security/threat-model.adoc`) but for the OIDC relying-party capability. Security is the centre of gravity: a sourced threat model and a sourced best-practices catalog, both drawn from the standards and current security research below.

== Guiding principle: a most-secure current subset

Security is the *leading* topic, not one section among many. This document set specifies the *smallest, most secure subset* of current OAuth 2.0 / OpenID Connect that an RP needs — it is *not* a full, spec-complete implementation.

* *Current only.* Adopt the latest recommended constructs (authorization code + PKCE, `iss`, PAR, DPoP/mTLS, the BCP RFC 9700, the OAuth 2.1 direction, FAPI 2.0 where it raises the bar).
* *Legacy/discouraged is out of scope by design.* Mechanisms the standards deprecate or discourage are deliberately *not* supported — and that exclusion is itself documented and sourced (below), so it reads as a decision, not an omission.
* *Security-driven requirements.* Requirements derive from the threat model: a feature is in scope only if it is required for secure operation, or is the secure way to do something. "The spec allows it" is not, by itself, a reason to include it.

== Deliberately excluded (legacy / discouraged)

Documented, sourced non-goals. The RP MUST NOT require or implement these; each exclusion is restated in the relevant aspect doc with its source.

* *Implicit grant* (`response_type=token` / `id_token token`) — RFC 9700 §2.1.2 (SHOULD NOT); removed in OAuth 2.1. Use code + PKCE.
* *Resource Owner Password Credentials grant* — RFC 9700 §2.4 (MUST NOT); removed in OAuth 2.1.
* *Hybrid flows* (`code id_token`, `code token`, …) — unnecessary for a secure RP and re-introduce front-channel token exposure. Use plain `code`.
* *PKCE `plain` method* — `S256` only (RFC 7636, RFC 9700).
* *`alg=none` / unsigned or weak-algorithm ID tokens* — RFC 8725 (JWT BCP). Require asymmetric signature verification.
* *Tokens in URL query/fragment* — RFC 9700 §4.3; use `form_post` and code exchange at the token endpoint.
* *Bare bearer tokens where sender-constraining is feasible* — prefer DPoP/mTLS (RFC 9449, FAPI 2.0); bearer only as a documented fallback.
* *Non-rotating long-lived refresh tokens for public clients* — RFC 9700 §4.14.
* *Front-channel logout / Session-Management iframe as the primary mechanism* — known reliability/security weaknesses; prefer back-channel logout (OIDC Back-Channel Logout 1.0). Front-channel is optional/secondary at most.
* *Unvalidated dynamic registration of `jwks_uri`/`*_uri`* — SSRF vector (CVE-2026-1180); excluded unless strictly validated.

== Organizing principle: separate the three aspects

The RP surface splits into three distinct concern areas. Keep them in separate documents and requirement groups — do not blend them.

[cols="1,3"]
|===
| Aspect | What it covers

| *Token* (largely "as of now")
| Token handling/validation. Mostly *inherited from the validation core* — the RP docs *reference* the existing validation requirements/threat model and add only the RP-specific delta: ID-token validation (`iss`/`aud`/`exp`/`nonce`/`azp`/`at_hash`/`c_hash`/signature), token storage, refresh-token rotation, sender-constrained tokens.

| *Process* (net-new)
| Protocol/flow orchestration: authorization-code + PKCE, `state`/CSRF, `nonce` binding, exact redirect-URI matching, mix-up defence, code injection, PAR, response mode, logout (RP-initiated / front- / back-channel), session management, clickjacking.

| *Fetch* (net-new)
| Server-side retrieval/HTTP: discovery (`.well-known`), JWKS fetch/cache/rotation, token-endpoint exchange, userinfo, TLS, *SSRF* on attacker-influenced URLs (`jwks_uri`, discovery), timeouts/resilience, response validation.
|===

The *token* aspect must not duplicate the validation docs — it links to them and states only what is new for an RP. *Process* and *fetch* are genuinely new and carry most of the new threat surface.

== Requirement-ID & sourcing discipline

* RP requirements use `TOKEN-SHERIFF-RP-N` (per Plan 02), *grouped by aspect* within the doc (Token / Process / Fetch sections).
* *Every* normative requirement and *every* threat entry cites a source with section anchor (RFC / OIDC spec / OWASP / FAPI). RFC 2119 keywords (MUST/SHOULD/MAY) trace back to the cited BCP.
* Requirements are *security-driven* — each traces to a threat it mitigates or is the secure way to perform a needed operation. Exclusions (the legacy/discouraged list) are sourced too.
* `references.adoc` holds the single bibliography; all docs cite into it.

== Execution

. *Scaffold + bibliography* — create the doc set skeleton and `references.adoc` from the source catalog below.
. *Requirements* (`requirements.adoc`) — author `TOKEN-SHERIFF-RP-N`, grouped by aspect, each sourced.
. *Threat model* (`threat-model.adoc`) — STRIDE/attack-catalog per aspect, each entry → source + mitigation + the requirement that covers it.
. *Best practices* (`best-practices.adoc`) — the sourced checklist below, normative.
. *Specifications* — `process-flow.adoc`, `fetch-discovery-jwks.adoc`, `token-handling.adoc`.
. *Traceability + verify* — cross-link requirements ↔ threats ↔ best practices ↔ spec; confirm aspect separation and full sourcing.

== Seed: threat catalog by aspect (with source anchors)

Minimum coverage for `threat-model.adoc`. Each → mitigation + covering requirement during authoring.

.Process aspect
* Insufficient redirect-URI validation — RFC 9700 §4.1 → exact string match
* Mix-up attacks (multiple AS) — RFC 9700 §4.4, RFC 9207 → `iss` validation or distinct redirect URIs per issuer
* Authorization-code injection — RFC 9700 §4.5 → PKCE (public clients MUST) and/or `nonce`
* PKCE downgrade — RFC 9700 §4.8 → verify AS PKCE support
* CSRF — RFC 9700 §4.7 → session-bound `state`, or PKCE, or `nonce`
* Open redirection — RFC 9700 §4.11 → no open redirectors
* Clickjacking — RFC 6819 §4.4.1.9 → CSP `frame-ancestors` / framebusting
* Logout/session integrity & session fixation — OIDC Front-/Back-Channel Logout, Session Management 1.0

.Token aspect
* Access-token leakage via Referer — RFC 9700 §4.2 → `Referrer-Policy`, code at token endpoint
* Token exposure in browser history — RFC 9700 §4.3 → no tokens in query; `form_post`
* Access-token injection — RFC 9700 §4.6 → `at_hash`; no implicit grant
* Implicit-grant token leakage — RFC 9700 §2.1.2 → code+PKCE only
* Refresh-token theft — RFC 9700 §4.14 → rotation (public clients) or sender-constraining
* Token replay / lack of sender-constraining — RFC 9700 §4.10, RFC 9449, FAPI 2.0 → DPoP or mTLS
* ID-token validation — OpenID Connect Core §3.1.3.7 → `iss`/`aud`/`exp`/`iat`/`nonce`/signature/`azp`

.Fetch aspect
* SSRF via `jwks_uri` / discovery URLs — OWASP, CVE-2026-1180 → host allow-listing, egress restriction, block internal/link-local ranges
* Malicious IdP endpoints advertised via discovery — RFC 8414 / OIDC Discovery → validate `issuer`, pin/validate metadata
* TLS downgrade / MITM on fetches — RFC 9700, RFC 8414 → enforce TLS, validate certificates
* DoS via large/slow metadata or JWKS responses → timeouts, size limits
* JWKS rotation/caching correctness (already handled in the validation core) → reference the core's strategy

== Seed: best-practices checklist by aspect (sourced)

Normative basis for `best-practices.adoc`.

* *Process:* authorization-code + PKCE for all client types; never the implicit grant (RFC 9700 §2.1.2, FAPI 2.0). Exact redirect-URI match (§4.1). CSRF via session-bound `state` / PKCE / `nonce` (§4.7). Mix-up defence via `iss` (RFC 9207, §4.4). Prefer PAR for request integrity (RFC 9126, FAPI 2.0). `form_post` response mode; no tokens in query (§4.3).
* *Token:* validate the ID token fully — `iss`/`aud`/`exp`/`iat`/`nonce`/signature/`azp`/`at_hash` (OIDC Core §3.1.3.7). Sender-constrain access tokens with DPoP or mTLS (RFC 9449, §4.10, FAPI 2.0). Rotate refresh tokens for public clients (§4.14). Short-lived access tokens; minimal token storage.
* *Fetch:* enforce TLS on all server-side fetches. Defend against SSRF on `jwks_uri`/discovery — allow-list hosts, restrict egress, block internal ranges (OWASP, CVE-2026-1180). Apply timeouts and response size limits. Reuse the validation core's JWKS cache/rotation.
