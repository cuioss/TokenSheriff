== Approach: protocol per protocol

Implement one *protocol/flow at a time* as a vertical slice ‚Äî production code + unit tests + integration tests ‚Äî not horizontally by layer. Why:

* Each protocol (grant, flow, endpoint) is a coherent unit that can be verified *end-to-end against a real OP*.
* Plan 04's threat model maps to *adversarial unit tests* (only a controllable mock can produce malicious responses).
* You ship working, tested capability incrementally ‚Äî no big-bang.

== Testing model: MockWebServer (unit) + real Keycloak (integration)

Two distinct layers ‚Äî do not conflate them:

* *Unit tests* ‚Äî in `token-sheriff-client` `src/test`, using *MockWebServer* to simulate the OP (discovery/JWKS/token/userinfo). Vital, but *unit*: this is where the *adversarial* coverage lives ‚Äî malicious metadata, SSRF, wrong `iss`, code injection, PKCE downgrade, tampered/replayed tokens ‚Äî because only a controllable mock can emit those. Use the CUI MockWebServer standards (`@ModuleDispatcher`, HTTPS/TLS, generator-driven *attack-database*).
* *Integration tests* ‚Äî Failsafe `*IT` against a *real Keycloak* (testcontainers / the existing Keycloak harness), verifying each protocol *actually interoperates* with a real OP end-to-end. Primary home: the replacement consumer `portal-authentication-oauth` (Plan 05). Protocols the portal login flow cannot exercise on their own (`client_credentials`, DPoP, token exchange) get a Keycloak-backed `*IT` harness in the Token-Sheriff repo.

The user-facing rule: *MockWebServer proves the mechanics and the attacks; Keycloak proves it really works.* Both are required per increment.

== Per-increment definition of done

* Production code in `token-sheriff-client`, reusing the `commons` transport and the validation pipeline.
* *Unit tests* (MockWebServer): protocol mechanics + every adversarial case the increment's threats demand.
* *Integration tests* (real Keycloak): the protocol's happy-path/conformance end-to-end.
* Quarkus wiring + extension smoke where the increment is user-configurable.
* *Traceability*: each `CLIENT-N` requirement ‚Üí code + test; each threat ‚Üí an adversarial unit test.
* Build green: `./mvnw -Ppre-commit clean verify` then `./mvnw clean install`.

== Increment order (dependency-driven)

[cols="1,2,3,2"]
|===
| # | Protocol / capability | Unit (MockWebServer ‚Äî incl. adversarial) | Integration (real Keycloak)

| 1 | *Client config + discovery* ‚Äî config (issuer, client id, auth method, scopes), resolve endpoints via `commons` discovery; TLS + SSRF guard (provided by `commons`)
| Altered metadata; SSRF to internal/link-local; non-TLS rejected
| Discovery against Keycloak's `.well-known`

| 2 | *`client_credentials`* ‚Äî obtain access token, validate via pipeline
| Error responses; tampered token rejected
| Real token from Keycloak, validated

| 3 | *Client authentication* ‚Äî `client_secret_*` ‚Üí `private_key_jwt` (RFC 7523) ‚Üí mTLS (RFC 8705)
| Weak-auth-where-stronger-available refused; secret-over-non-TLS refused
| Each method accepted by Keycloak

| 4 | *`refresh_token`* ‚Äî refresh, rotation, revoke-on-misuse
| Rotation reuse-detection; expired
| Refresh cycle against Keycloak

| 5 | *`authorization_code` + PKCE (S256)* ‚Äî BFF flow: `state`/`nonce`/exact redirect-URI, code exchange, ID-token validation; *step-up* (`acr_values`/`max_age` + `insufficient_user_authentication` challenge, RFC 9470)
| Code injection; PKCE downgrade; `state`/`nonce` mismatch; wrong `iss`; step-up challenge ignored/forged
| Full login + code exchange via Keycloak; assert `insufficient_user_authentication`, retry with `acr_values` / `max_age`, and verify resulting `acr` / `auth_time`

| 6 | *Mix-up defence (`iss`, RFC 9207) + PAR (RFC 9126)*
| `iss` mismatch rejected
| PAR + `iss` against Keycloak (if supported)

| 7 | *userinfo* ‚Äî fetch + bind `sub`, correlate with ID token
| Tampered / `sub` mismatch rejected
| Userinfo from Keycloak

| 8 | *Sender-constraining ‚Äî DPoP (RFC 9449)* (+ mTLS-bound)
| Proof replay / `jti` reuse rejected
| DPoP-bound token against Keycloak

| 9 | *Token lifecycle + RP-initiated logout* ‚Äî server-side storage, refresh scheduling, revocation; `end_session_endpoint` trigger (`id_token_hint`/`post_logout_redirect_uri`/`state`)
| Lifecycle transitions; storage isolation; logout open-redirect via unmatched `post_logout_redirect_uri`, missing `id_token_hint`/`state`, stale token after logout (must be revoked)
| Revocation + RP-initiated logout against Keycloak (`id_token_hint`, exact-matched `post_logout_redirect_uri`, `state`)

| 10 | *Hardening sweep* ‚Äî verify the `commons` transport (incl. the relocated `wellknown`/`jwks`) is in use; close remaining `CLIENT-N` threat-model gaps (transport hardening itself is owned by `CMN-N`, Plan 03)
| Full attack-database across all fetch/retrieval paths
| ‚Äî

| 11 | *E2E replacement* ‚Äî wire `token-sheriff-client` into `portal-authentication-oauth` (own package)
| ‚Äî
| Keycloak login + token + userinfo + refresh + logout, as the legacy module did
|===

Increments 1‚Äì2 prove the whole retrieve‚Üívalidate loop; 5 is the BFF core; 11 closes the replacement loop.
