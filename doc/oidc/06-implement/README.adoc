== Approach: protocol per protocol

Implement one *protocol/flow at a time* as a vertical slice тАФ production code + unit tests + integration tests тАФ not horizontally by layer. Why:

* Each protocol (grant, flow, endpoint) is a coherent unit that can be verified *end-to-end against a real OP*.
* Plan 04's threat model maps to *adversarial unit tests* (only a controllable mock can produce malicious responses).
* You ship working, tested capability incrementally тАФ no big-bang.

== Testing model: MockWebServer (unit) + real Keycloak (integration)

[NOTE]
====
The *authoritative test specification* for this plan is
xref:../../client/specification/test-strategy.adoc[Client тАФ Test strategy] (under the
capability doc set). It owns the two-layer model, the dispatcher / attack-database adversarial
coverage, the *happy / corner-edge / attack* integration case classes, and the full
`CLIENT-N` тЖТ threat тЖТ unit тЖТ integration *traceability matrix*. This plan *executes* that
specification increment by increment; the summary below is the execution view тАФ the spec is
the contract.
====

Two distinct layers тАФ do not conflate them:

* *Unit tests* тАФ in `token-sheriff-client` `src/test`, using *MockWebServer* to simulate the OP (discovery/JWKS/token/userinfo/тАж), driven by the dispatcher artifact *extended in xref:../03-commons/README.adoc[Plan 03]* (`TokenDispatcher`, `UserInfoDispatcher`, тАж alongside the existing `WellKnownDispatcher`/`JwksResolveDispatcher`). Vital, but *unit*: this is where the *adversarial* coverage lives тАФ malicious metadata, SSRF, wrong `iss`, code injection, PKCE downgrade, tampered/replayed tokens тАФ because only a controllable mock can emit those. Use the CUI MockWebServer standards (`@ModuleDispatcher`, HTTPS/TLS, generator-driven *attack-database*).
* *Integration tests* тАФ Failsafe `*IT` against a *real Keycloak* (testcontainers / the existing Keycloak harness) *in the Token-Sheriff repo*, verifying each protocol *actually interoperates* with a real OP end-to-end. Every protocol тАФ `client_credentials`, DPoP, auth-code, token exchange тАФ gets its Keycloak-backed `*IT` here. (The eventual `portal-authentication-oauth` replacement is a *design reference* and *future work beyond these plans* тАФ see xref:../04-client-spec/README.adoc[Plan 04]; it is **not** an integration-test home for this plan.)

The user-facing rule: *MockWebServer proves the mechanics and the attacks; Keycloak proves it really works.* Both are required per increment.

== Per-increment definition of done

_The test-side rows below trace to the per-increment definition of done in
xref:../../client/specification/test-strategy.adoc[Test strategy]._

* Production code in `token-sheriff-client`, reusing the `commons` transport and the validation pipeline.
* *Unit tests* (MockWebServer): protocol mechanics + every adversarial case the increment's threats demand.
* *Integration tests* (real Keycloak): the protocol's happy-path/conformance end-to-end.
* Quarkus wiring + extension smoke where the increment is user-configurable.
* *Traceability*: each `CLIENT-N` requirement тЖТ code + test; each threat тЖТ an adversarial unit test.
* Build green: `./mvnw -Ppre-commit clean verify` then `./mvnw clean install`.

== Increment order (dependency-driven)

[cols="1,2,3,2"]
|===
| # | Protocol / capability | Unit (MockWebServer тАФ incl. adversarial) | Integration (real Keycloak)

| 1 | *Client config + discovery* тАФ config (issuer, client id, auth method, scopes), resolve endpoints via `commons` discovery; TLS + SSRF guard (provided by `commons`)
| Altered metadata; SSRF to internal/link-local; non-TLS rejected
| Discovery against Keycloak's `.well-known`

| 2 | *`client_credentials`* тАФ obtain access token, validate via pipeline
| Error responses; tampered token rejected
| Real token from Keycloak, validated

| 3 | *Client authentication* тАФ `client_secret_*` тЖТ `private_key_jwt` (RFC 7523) тЖТ mTLS (RFC 8705)
| Weak-auth-where-stronger-available refused; secret-over-non-TLS refused
| Each method accepted by Keycloak

| 4 | *`refresh_token`* тАФ refresh, rotation, revoke-on-misuse
| Rotation reuse-detection; expired
| Refresh cycle against Keycloak

| 5 | *`authorization_code` + PKCE (S256)* тАФ BFF flow: `state`/`nonce`/exact redirect-URI, code exchange, ID-token validation; *step-up* (`acr_values`/`max_age` + `insufficient_user_authentication` challenge, RFC 9470)
| Code injection; PKCE downgrade; `state`/`nonce` mismatch; wrong `iss`; step-up challenge ignored/forged
| Full login + code exchange via Keycloak; assert `insufficient_user_authentication`, retry with `acr_values` / `max_age`, and verify resulting `acr` / `auth_time`

| 6 | *Mix-up defence (`iss`, RFC 9207) + PAR (RFC 9126)*
| `iss` mismatch rejected
| PAR + `iss` against Keycloak (if supported)

| 7 | *userinfo* тАФ fetch + bind `sub`, correlate with ID token
| Tampered / `sub` mismatch rejected
| Userinfo from Keycloak

| 8 | *Sender-constraining тАФ DPoP (RFC 9449)* (+ mTLS-bound)
| Proof replay / `jti` reuse rejected
| DPoP-bound token against Keycloak

| 9 | *Token lifecycle + RP-initiated logout* тАФ server-side storage, refresh scheduling, revocation; `end_session_endpoint` trigger (`id_token_hint`/`post_logout_redirect_uri`/`state`)
| Lifecycle transitions; storage isolation; logout open-redirect via unmatched `post_logout_redirect_uri`, missing `id_token_hint`/`state`, stale token after logout (must be revoked)
| Revocation + RP-initiated logout against Keycloak (`id_token_hint`, exact-matched `post_logout_redirect_uri`, `state`)

| 10 | *Hardening sweep* тАФ verify the `commons` transport (incl. the relocated `wellknown`/`jwks`) is in use; close remaining `CLIENT-N` threat-model gaps (transport hardening itself is owned by `COMMONS-N`, Plan 03)
| Full attack-database across all fetch/retrieval paths
| тАФ

| 11 | *Full-flow E2E (in-repo)* тАФ the complete confidential-client flow wired together in the Token-Sheriff integration tests
| тАФ
| Keycloak login + token + userinfo + refresh + logout, end-to-end
|===

Increments 1тАУ2 prove the whole retrieveтЖТvalidate loop; 5 is the BFF core; 11 ties the full flow together end-to-end.
