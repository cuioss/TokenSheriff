== Approach: protocol per protocol

Implement one *protocol/flow at a time* as a vertical slice ŌĆö production code + unit tests + integration tests ŌĆö not horizontally by layer. Why:

* Each protocol (grant, flow, endpoint) is a coherent unit that can be verified *end-to-end against a real OP*.
* Plan 04's threat model maps to *adversarial unit tests* (only a controllable mock can produce malicious responses).
* You ship working, tested capability incrementally ŌĆö no big-bang.

== Testing model: MockWebServer (unit) + real Keycloak (integration)

Two distinct layers ŌĆö do not conflate them:

* *Unit tests* ŌĆö in `token-sheriff-client` `src/test`, using *MockWebServer* to simulate the OP (discovery/JWKS/token/userinfo/ŌĆ”), driven by the dispatcher artifact *extended in xref:../03-commons/README.adoc[Plan 03]* (`TokenDispatcher`, `UserInfoDispatcher`, ŌĆ” alongside the existing `WellKnownDispatcher`/`JwksResolveDispatcher`). Vital, but *unit*: this is where the *adversarial* coverage lives ŌĆö malicious metadata, SSRF, wrong `iss`, code injection, PKCE downgrade, tampered/replayed tokens ŌĆö because only a controllable mock can emit those. Use the CUI MockWebServer standards (`@ModuleDispatcher`, HTTPS/TLS, generator-driven *attack-database*).
* *Integration tests* ŌĆö Failsafe `*IT` against a *real Keycloak* (testcontainers / the existing Keycloak harness) *in the Token-Sheriff repo*, verifying each protocol *actually interoperates* with a real OP end-to-end. Every protocol ŌĆö `client_credentials`, DPoP, auth-code, token exchange ŌĆö gets its Keycloak-backed `*IT` here. (The eventual `portal-authentication-oauth` replacement is a *design reference* and *future work beyond these plans* ŌĆö see xref:../04-client-spec/README.adoc[Plan 04]; it is **not** an integration-test home for this plan.)

The user-facing rule: *MockWebServer proves the mechanics and the attacks; Keycloak proves it really works.* Both are required per increment.

== Per-increment definition of done

* Production code in `token-sheriff-client`, reusing the `commons` transport and the validation pipeline.
* *Unit tests* (MockWebServer): protocol mechanics + every adversarial case the increment's threats demand.
* *Integration tests* (real Keycloak): the protocol's happy-path/conformance end-to-end.
* Quarkus wiring + extension smoke where the increment is user-configurable.
* *Traceability*: each `CLIENT-N` requirement ŌåÆ code + test; each threat ŌåÆ an adversarial unit test.
* Build green: `./mvnw -Ppre-commit clean verify` then `./mvnw clean install`.
