== Goal

Stand up the complete skeleton the RP work will fill: the engine module, a Quarkus extension for it, the build assembly (BOM + reactor), and an integration-test home. After Plan 04 a developer only adds content (per xref:../03-rp-spec/README.adoc[Plan 03]); no structural work remains.

== Decision: standalone core-like engine + a dedicated RP Quarkus extension

. *Engine — standalone, modelled on `token-sheriff-core`.* Pure Java, HTTP via `cui-http`, depends on `token-sheriff-core`. *Not* inside the Quarkus extension.
+
Why: the primary replacement target `portal-authentication-oauth` is *CDI / RESTEasy / portal, not Quarkus* — the engine must be framework-agnostic. It mirrors the proven `core` + `quarkus` split, and the core is already pure Java + `cui-http` (the RP's HTTP foundation). Opt-in stays clean (validation-only users pull no RP code).

. *Quarkus — a new extension that depends on the existing one.* A dedicated `token-sheriff-rp-quarkus` extension (runtime + deployment), depending on the existing `token-sheriff-quarkus` extension *and* the engine.
+
Why a separate extension rather than folding RP config into the existing one: it preserves opt-in (Quarkus validation-only users don't pull the RP engine), and it gives the RP its own integration-test surface. It depends on the existing extension because RP reuses the validation CDI wiring (the `TokenValidator` etc.) and adds RP producers on top.

. *Verification is integration-test-driven.* RP behaviour (token retrieval, flows, fetching) is exercised end-to-end, not by unit tests alone.

. *RP integration tests live in their own package inside the existing IT module* — `token-sheriff-quarkus-integration-tests` — not a new module. (Own package, not own module — per preference.)

The portal adapter (preserving `Oauth2Service` / `AuthenticatedUserInfo`) lives in `cui-portal-core`, not here.

== Deliverables (the four pieces)

| Piece | What lands in Plan 04 (no functional content)

| `token-sheriff-rp` — top-level, core-like. `package-info.java` in `de.cuioss.sheriff.token.rp`; deps `token-sheriff-core` + `cui-http`.

| *extension*
| `token-sheriff-rp-quarkus` (runtime) + `token-sheriff-rp-quarkus-deployment`, under `token-sheriff-quarkus-parent`. Minimal Quarkus extension that registers a `token-sheriff-rp` feature and nothing else. Deps: existing `token-sheriff-quarkus`(`-deployment`) + `token-sheriff-rp`.

| *assembly*
| Build wiring that assembles the set into a releasable whole: root `<modules>`, `token-sheriff-quarkus-parent` `<modules>`, and BOM entries for the consumer-facing artifacts (`token-sheriff-rp`, `token-sheriff-rp-quarkus`).

| *it*
| An RP integration-test *package* inside `token-sheriff-quarkus-integration-tests` (own package), depending on the RP extension, with a *smoke test* that the app boots with the RP extension present — verifying the assembly, exercising no RP behaviour yet.
|===

== Wiring checklist ("link it everywhere")

* `pom.xml` (root) — add `<module>token-sheriff-rp</module>`.
* `token-sheriff-quarkus-parent/pom.xml` — add `<module>token-sheriff-rp-quarkus</module>` and `…-deployment`.
* `bom/pom.xml` — add `token-sheriff-rp` and `token-sheriff-rp-quarkus` (the BOM enumerates consumer-facing artifacts).
* New POMs — model the engine on `token-sheriff-core`; model the RP extension pair on the existing `token-sheriff-quarkus`(`-deployment`), adding the dependency on the existing extension + engine.
* RP extension descriptor — `quarkus-extension.yaml` (name "Token-Sheriff RP"); deployment `*Processor` registering the `token-sheriff-rp` feature.
* `token-sheriff-quarkus-integration-tests/pom.xml` — add a dependency on `token-sheriff-rp-quarkus`; create the RP IT package + a boot/smoke `*IT`.
* Source skeletons — `package-info.java` for each new package; `doc/` stub for the engine per xref:../02-doc-structure/README.adoc[Plan 02].
* CI/quality — reactor builds the new modules automatically; confirm `.github/project.yml` (Sonar) scans them and the empty modules don't fail the gate; `.github/dependabot.yml` Maven root covers them.
* Docs/README — list the new module + extension.

== "Builds empty" concern

Empty modules can fail quality gates (JaCoCo coverage on zero classes, PMD, Sonar new-code coverage). Resolve as wiring:

* Keep only `package-info.java` (+ the minimal extension feature class) so license/checkstyle pass.
* Ensure coverage gates treat near-empty modules as N/A (or scope them out of the ratio until they have content). The smoke IT gives the extension a first executed path.
* `./mvnw -Ppre-commit clean verify` and `./mvnw clean install` must both be green before the skeleton is "done".
