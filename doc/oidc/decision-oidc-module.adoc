== Context

The library is, today, a *passive multi-issuer JWT validation library*. It deliberately stops at "validate, never acquire" — the README states it "will never create tokens, only validate them."

A survey of the validation module confirms it already covers the *validation-side* of OpenID Connect:

* OIDC Discovery — a dedicated `wellknown` package (`HttpWellKnownResolver`, `WellKnownConfig`, `WellKnownConfigurationConverter`)
* JWKS retrieval — a `jwks.http` package (fetch, cache, rotation)
* ID token validation — `IdTokenValidationPipeline`, `IdTokenContent`, `IdTokenRequest`
* No userinfo endpoint, authorization-code/PKCE flow, token-endpoint exchange, session, or logout (0 references in the validation module)

Two distinct additions are in view:

. The *active* OIDC capabilities — the client (relying-party) side: authorization-code / PKCE flows, token-endpoint exchange, userinfo, `nonce`/`state`. (Secure subset scoped in xref:04-client-spec/README.adoc[Plan 04].)
. A realization that a whole *class of base-layer concerns* belongs to neither validation nor client but to *both*: the outbound IdP HTTP (discovery + JWKS today, token/userinfo/… tomorrow), the *error model* (typed exceptions → RFC 9457 at the HTTP edge), *security events* (`SecurityEventCounter`), and *metrics*. Today these sit in validation; they should be a shared base so the client reuses them rather than duplicating them.

The question is **where** this code lives: a separate Git project (the original "OIDC-Sheriff" idea) or modules inside this repository — and how many modules.

== Decision

Keep everything in **this multi-module repository** (no separate Git project) and split into **two Maven modules** — with the shared base as an *ArchUnit-enforced package layer*, not a third artifact:

[source]
----
token-sheriff-validation                 (one artifact, two enforced layers)
  de.cuioss.sheriff.token.commons.*     base: transport / error / events / metrics   (deps: cui-http, cui-tooling)
        ▲   ArchUnit: commons must NOT depend on …validation..
  de.cuioss.sheriff.token.validation.*  token validation (signatures, claims, pipeline)
        ▲   Maven
token-sheriff-client                      flows / token acquisition / lifecycle        (deps: validation)
----

* **`commons` — a package layer inside `token-sheriff-validation`** (`de.cuioss.sheriff.token.commons.*`): the shared base — *transport* (discovery, JWKS, token, userinfo, revocation, introspection, end-session, PAR; TLS, SSRF defence, resilience, IdP-error normalization), the *error model* (typed exceptions the HTTP edges map to RFC 9457), *security events*, and *metrics*. Depends on `cui-http` and `cui-tooling` only; ArchUnit forbids it depending "up" on `…validation..`.
* **`token-sheriff-validation`** — the artifact; ships the `commons` base packages + the validation packages. Today's `wellknown`/`jwks.http`/`SecurityEventCounter`/metric identifiers are re-packaged into the `commons` layer (xref:03-commons/README.adoc[Plan 03]).
* **`token-sheriff-client`** — a *separate* module/artifact: flows / token acquisition / lifecycle. Depends on `validation` (which transitively provides the `commons` base).

*Why no separate `commons` artifact:* **no consumer wants the base without validation** — the client validates the tokens it retrieves, so it pulls `validation` anyway and gets the base transitively; `nifi-extensions` wants `validation`. A third artifact would add a POM/release/BOM surface for a dependency nobody picks alone. ArchUnit gives the layering guarantee (base stays dependency-light, never depends up) without the packaging overhead. `commons` keeps a *doc-level* identity (`doc/commons/`, `COMMONS-N`).

The seams are *not* "OAuth vs OIDC" (validation already does OIDC discovery and ID tokens). They are:

* **Shared base — transport, error model, events, metrics → `commons` packages** (hardened substrate; never interprets trust)
* **Passive validation of tokens → `validation` packages**
* **Active acquisition / flows → `client` module**

== Rationale

The stated goal of separation — "users choose the dependency that matches their requirements" — is satisfied by a separate *artifact*. It does **not** require a separate *repository*. A second repo only buys a repo boundary; it costs a version-compatibility matrix, two release trains, duplicated CI/SonarCloud/release/BOM infrastructure, and cross-repo pull requests every time the seam shifts.

The dependency direction is the natural, clean layering — `client → validation`, with `commons` the enforced base layer *inside* `validation` — and never reverses. A consumer that needs only validation (e.g. `nifi-extensions`) pulls `validation`; a consumer that needs acquisition adds `client`.

The validation module's public API is still pre-1.0 (Maven Central `v0.8.x`) and actively evolving (a recent 1.0 cleanup renamed `well_known → wellknown`, etc.). The client layer will exercise validation-module APIs in ways that are hard to predict; that co-evolution is *trivial in one repo* and *painful across two*.

The asymmetry is decisive: **promoting a coupled layer later is cheap** — `commons` packages → their own module (move the `…commons.*` tree, swap the ArchUnit rule for a Maven dependency), or a module → its own repo (move the directory, redirect the dependency); **merging back after a premature split is expensive.** Start coupled; split on evidence.

== Consequences

=== Positive

* A separate, opt-in artifact — the user-facing separation of concerns is fully delivered.
* Shared CI, SonarCloud, BOM, release automation, and benchmarking — no duplication.
* Atomic cross-cutting changes across the validation/client seam (no version matrix).
* The client layer can co-evolve with the still-moving validation-module API during the pre-1.0 window.

=== Negative / trade-offs

* One repository grows larger (mitigated: it is already multi-module).
* *Scope/identity shift.* The validation module promises "validate, never acquire." Housing token-acquiring client code under a *distinct artifact and name* preserves that promise; it must not leak into `token-sheriff-validation`.

== When to promote `token-sheriff-client` to its own repository

Revisit this decision — and split — when **any** of the following becomes true:

* An **independent release cadence** is genuinely blocked by shared versioning.
* The client layer drags in a **heavier or higher-risk dependency tree** that should not touch the validation module's supply-chain surface.
* **Different audience, governance, or license** (e.g., the client moves under a different commercial tier).
* A **different maintainer set** with a separate review/security process.

Until one of these bites, the module is strictly better.

== When to promote `commons` to its own module

Keep it a package layer until **any** of these becomes true; then move `…commons.*` into a `token-sheriff-commons` artifact and swap the ArchUnit rule for a Maven dependency:

* A consumer genuinely needs the **base without validation** (e.g. an OAuth-client use that does not run the validation pipeline).
* The base must ship on an **independent release cadence** from validation.
* The base grows a **dependency tree** that validation-only consumers should not inherit.

Until then, ArchUnit delivers the same layering guarantee with one fewer artifact.

== Resolved by later plans

* *Commons base layer* — the `de.cuioss.sheriff.token.commons.*` package layer inside `token-sheriff-validation`, the re-packaging of `wellknown`/`jwks.http` + `SecurityEventCounter` + metric identifiers into it, the error model, the ArchUnit rules enforcing the layering, and its requirements/spec/threat-model under `doc/commons/`. See xref:03-commons/README.adoc[Plan 03].
* *Module/artifact name* — `token-sheriff-client` (engine), `token-sheriff-client-quarkus` (extension). See xref:05-client-module/README.adoc[Plan 05].
* *Quarkus integration* — a dedicated `token-sheriff-client-quarkus` extension depending on the existing one (not folded in). See xref:05-client-module/README.adoc[Plan 05].
* *API surface / first flows* — secure-subset scope and the protocol-by-protocol order (`client_credentials` first, then auth-code + PKCE) are in xref:04-client-spec/README.adoc[Plan 04] and xref:06-implement/README.adoc[Plan 06].

== Still open

* Session/state storage abstraction and its dependency footprint (addressed in the Plan 06 token-lifecycle increment).

== Alternatives considered

* **Separate Git project (original "OIDC-Sheriff").** Rejected for now: same user-facing benefit as a module, but with cross-repo maintenance overhead and a version-compatibility matrix during the formative pre-1.0 period. Kept as a documented future option above.
* **Fold the client into `token-sheriff-validation`.** Rejected: breaks the validation module's "validate, never acquire" promise and bloats the most security-critical, most-reused artifact.
