== Goal

Stand up the complete in-repo skeleton the client work will fill: the engine module, a Quarkus extension for it, and the build assembly (BOM + reactor). After Plan 05 a developer only adds content (per xref:../04-client-spec/README.adoc[Plan 04]); no structural work remains.

[NOTE]
.Execution decision ŌĆö transport migration folded in (option b)
====
xref:../03-commons/README.adoc[Plan 03] deferred the transport re-packaging (`ŌĆ”validation.wellknown` + `ŌĆ”validation.jwks.http` ŌåÆ `ŌĆ”commons.transport`, plus the log-registry split) onto this plan. Two options were on the table: *(a)* keep Plan 05 a pure empty skeleton and defer the transport migration to a dedicated PR ahead of Plan 06; *(b)* fold the transport migration into Plan 05.

*Chosen: (b)* ŌĆö the transport migration was folded in. Within (b), the `HttpJwksLoader Ōåö JWKS-crypto` coupling was *inverted/split* (not carried into commons): the pure HTTP/JSON transport plumbing moved to `commons.transport`, while the JWKS crypto orchestrator (`HttpJwksLoader implements JwksLoader`, `JWKSKeyLoader`, `KeyInfo`, `jwks.parser/key`, `security`) stayed in `validation` and now depends *down* on `commons.transport`. This honors the Plan 03 seam ("commons moves bytesŌĆ” never interprets a token's trust") and satisfies the `CommonsLayeringTest` ArchUnit boundary. See the xref:../03-commons/README.adoc#_verification[Plan 03 verification] entry for the full type inventory.
====

== Decision: standalone engine, dedicated Quarkus extension

. *Engine ŌĆö standalone, modelled on `token-sheriff-validation`.* Pure Java, depends on `token-sheriff-validation` ŌĆö which provides both the `commons` base layer (transport, error model, events, metrics) and token validation. *Not* inside the Quarkus extension.
+
Why: the replacement target `portal-authentication-oauth` is *CDI / RESTEasy / portal, not Quarkus* ŌĆö the engine must be framework-agnostic. It mirrors the proven `validation` + `quarkus` split; the transport foundation is `commons` (which wraps `cui-http`), so the client adds flows, not HTTP. Opt-in stays clean.

. *Quarkus ŌĆö a new extension that depends on the existing one.* `token-sheriff-client-quarkus` (runtime + deployment), depending on the existing `token-sheriff-validation-quarkus` extension *and* the engine. A separate extension (not folded in) preserves opt-in and reuses the existing validation CDI wiring. Verified by the build assembly.

Why no integration-test piece here: `portal-authentication-oauth` is the *replacement target* that motivates the framework-agnostic engine (above) ŌĆö a design reference, not a deliverable. Wiring the engine into the portal is *future work beyond Plans 01ŌĆō06*; the in-repo client integration tests (real Keycloak) belong to xref:../06-implement/README.adoc[Plan 06]. This plan stays *in-repo* and structure-only.

* `./mvnw -Ppre-commit clean verify` and `./mvnw clean install` must both be green (in each repo) before the skeleton is "done".
