== Context

The name "OAuth" mislabels what this project is. It is a *multi-issuer JWT validation* library that already performs OpenID Connect Discovery and validates ID tokens — and the planned xref:decision-oidc-module.adoc[OIDC relying-party module] will widen scope further.

Critically, "oauth" is not only in the repository name. It is baked into three layers with very different change costs, and the project is still **pre-1.0 (Maven Central `v0.8.x`)**:

[cols="2,2,1"]
|===
| Layer | Cost to change | Breaks users?

| GitHub description + topics
| Trivial
| No

| GitHub repo name (`OAuthSheriff`)
| Cheap (GitHub redirects) — but ripples into badges, GitHub Pages URL, SonarCloud project key, OpenSSF entry, and all doc links
| No

| Maven `groupId` `de.cuioss.sheriff.oauth` + `oauth-sheriff-*` artifactIds + Java packages `de.cuioss.sheriff.oauth.*`
| Real migration
| **Yes**
|===

The choice is effectively binary, because the *half-measure* — renaming the repo while the coordinates still say `…sheriff.oauth.*` — is **more** confusing than either pure state.

== Considered & rejected: Strategy A — keep the name as a brand, fix the description now

_Superseded by Option B (see the decision-update note above). Retained for the record._

* **Keep** the GitHub repository name `OAuthSheriff` and the published Maven coordinates / package names **unchanged**. Treat "Sheriff" as the brand and "OAuth" as a historical brand token (cf. Keycloak, Dex, Ory Hydra — none literal).
* **Fix the description and topics now** — the genuinely misleading, zero-breakage surface where users search and judge.
* **Defer** any full rebrand (coordinates + packages) to a *deliberate 1.0 decision* (see "Option B" below), not as a side effect of the OIDC discussion.

=== Concrete changes (zero breakage)

* *Description:* `Comprehensive multi-issuer JWT validation for Java — offline OAuth2 / OpenID Connect token validation with OIDC Discovery.`
* *Topics:* add `oidc`, `openid-connect`, `jwt-validation`, `multi-issuer`, `quarkus`; keep `java`, `jwt`, `offline-validation`. `oauth2` stays — the library *does* validate OAuth2 JWTs.

== Rationale

The expensive, genuinely-misleading surface is the *topics and description*, which are free to fix. The mascot, the muscle memory, and the published coordinates all stay intact — zero breakage right before 1.0. This delivers a separate-repo-grade benefit (an accurate public identity) at description-grade cost.

== Rejected alternatives

* **Half-rename (repo only).** Rejected: leaves `de.cuioss.sheriff.oauth.*` coordinates and packages saying "oauth" — inconsistency tax, worse than either pure state.
* **Keep everything as-is.** Rejected: the `oauth` / `oauth2` topics and the name actively misinform about scope.
== Chosen: Option B — full rebrand to Token-Sheriff

Do a *coherent* rebrand across all surfaces at once, while pre-1.0 is still the cheap window (after 1.0 it is a breaking migration consumers would resent).

Name selection (must reflect the enduring core — multi-issuer JWT validation — not a subset/overstatement):

* **Token-Sheriff** — *chosen.* Accurate today and roomy enough for the future RP module (tokens, not only validation).
* **JWT-Sheriff** — considered; slightly narrow once the RP module lands.
* Rejected: `Auth-Sheriff` (vague: authn vs authz), `OIDC-Sheriff` (reserve for the RP; overstates today), keeping `OAuth-*` (the actual inaccuracy).

Verified free on Maven Central (0 hits), GitHub, and npm; only loose adjacency is the unrelated "Crypto Sheriff" / bare "Sheriff" tools.

The full target scheme, blast radius, Maven Central relocation mechanism, and execution phases live in the xref:../rebrand/plan-token-sheriff-rebrand.adoc[Token-Sheriff Rebrand Plan]. Consumer steps are in the xref:../rebrand/migration-nifi-extensions.adoc[nifi-extensions migration guide].
