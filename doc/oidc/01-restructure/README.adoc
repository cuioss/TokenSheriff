== Goal

Rename *OAuth-Sheriff* to *Token-Sheriff*. The name reflects the enduring scope — multi-issuer JWT validation, with headroom for the planned xref:../decision-oidc-module.adoc[OIDC client module]. "Token-Sheriff" is unused on Maven Central, GitHub, and npm.

== Target naming scheme

The `de.cuioss.sheriff.*` / `sheriff.*` roots stay stable; only the `oauth → token` leaf changes, minimising consumer churn.

[cols="2,3,3"]
|===
| Surface | From | To

| GitHub repo            | `cuioss/OAuthSheriff`            | `cuioss/TokenSheriff`
| Display / brand        | OAuth-Sheriff                   | Token-Sheriff
| Maven groupId          | `de.cuioss.sheriff.oauth`       | `de.cuioss.sheriff.token`
| Artifacts              | `oauth-sheriff-*`               | `token-sheriff-*`
| Java packages          | `de.cuioss.sheriff.oauth.*`     | `de.cuioss.sheriff.token.*`
| Config property prefix | `sheriff.oauth.*`               | `sheriff.token.*`
| Metrics prefix         | `sheriff.oauth.*`               | `sheriff.token.*`
| Requirement IDs        | `OAUTH-SHERIFF-N`               | `VAL-N` (shortened; client capability adds `CLIENT-N` later, see xref:../02-doc-structure/README.adoc[Plan 02])
| Quarkus feature / class | feature `oauth-sheriff`, `OAuthSheriffProcessor` | feature `token-sheriff`, `TokenSheriffProcessor`
| SonarCloud project key | `cuioss_OAuth-Sheriff`          | `cuioss_TokenSheriff`
| GitHub Pages base      | `cuioss.github.io/OAuth-Sheriff` | `cuioss.github.io/TokenSheriff`
|===

The mascot (`doc/resources/Sheriff.png`) and the "Sheriff" brand are unchanged — only the qualifier changes. There are no `module-info.java` files, so no JPMS module names to migrate.

Modules are renamed with *capability* names (not `-core`): `oauth-sheriff-core → token-sheriff-validation` (package `…token.validation`), `oauth-sheriff-quarkus → token-sheriff-validation-quarkus` (and `-deployment`). Aggregator/test modules keep the neutral form: `oauth-sheriff-quarkus-parent → token-sheriff-quarkus-parent`, `…-integration-tests → token-sheriff-quarkus-integration-tests`. (This plan only *renames*; the wider three-module architecture is added later — the `token-sheriff-commons` base module (transport + error model + events + metrics) is extracted from validation by xref:../03-commons/README.adoc[Plan 03], and the client modules by xref:../05-client-module/README.adoc[Plan 05].)

== Tooling

The Java type/package and Maven-coordinate refactoring is done with *OpenRewrite*, not text substitution: it is AST-aware, relocates source files to the new package directories, scopes to real package references (not the substring "oauth" in prose or jar names), is idempotent, and yields a reviewable diff. This repository already runs OpenRewrite (`cui-rewrite`). Text-only surfaces use targeted find/replace.

[cols="2,3"]
|===
| Surface | Recipe / tool

| Java packages, imports, FQNs, Javadoc `@link`, file relocation | `org.openrewrite.java.ChangePackage`
| Renamed types (`OAuthSheriffProcessor` → `TokenSheriffProcessor`, log-constant holders) | `org.openrewrite.java.ChangeType`
| Maven dependency + managed-dependency coordinates | `org.openrewrite.maven.ChangeDependencyGroupIdAndArtifactId`, `ChangeManagedDependencyGroupIdAndArtifactId`
| Parent POM coordinates | `org.openrewrite.maven.ChangeParentPom`
| Config / metric key literals + `.properties` keys (`sheriff.oauth` → `sheriff.token`) | `org.openrewrite.properties.ChangePropertyKey` + `org.openrewrite.text.FindAndReplace`
| Requirement IDs `OAUTH-SHERIFF-N`, docs, badges, workflows, README `alt` | `org.openrewrite.text.FindAndReplace` / scripts
|===

A module's *own* `<groupId>`/`<artifactId>` has no first-class recipe; those ~6 poms (`token-sheriff-parent`, `-validation`, `-validation-quarkus`, `bom`) are changed via IDE or a scoped `text.FindAndReplace`. Compose the recipes into one `rewrite.yml` and run `./mvnw rewrite:run`. Publish the same composite as a named recipe (`de.cuioss.sheriff.token.MigrateFromOAuthSheriff`) so consumers migrate with one command.

== Execution

=== Phase 0 — Prerequisites
* Cut the rebrand as version `0.9.0` (the rename/refactor release; `1.0.0` is reserved for the implemented-and-tested client module).
* Snapshot current published coordinates/versions (needed for relocation stubs).

=== Phase 1 — In-repo rename (single atomic branch)

*1a — Structural, via OpenRewrite* (`./mvnw rewrite:run`):

* `ChangePackage` `de.cuioss.sheriff.oauth` → `de.cuioss.sheriff.token` (relocates files, fixes imports/FQNs/Javadoc).
* `ChangeType` for renamed classes (`OAuthSheriffProcessor` → `TokenSheriffProcessor`, log-constant holders).
* `ChangeParentPom` + `ChangeManagedDependencyGroupIdAndArtifactId` + `ChangeDependencyGroupIdAndArtifactId` for inter-module coordinates and the BOM.

*1b — Module own-coordinates & metadata* (IDE / scoped text):

* Each module's own `<groupId>` → `de.cuioss.sheriff.token`, `<artifactId>` `oauth-sheriff-* → token-sheriff-*`, plus `<name>`/`<url>`.
* Rename module directories `oauth-sheriff-* → token-sheriff-*`.
* `quarkus-extension.yaml`, Quarkus feature string, log constants.

*1c — Text surfaces* (`text.FindAndReplace` / `properties.ChangePropertyKey`):

* Config + metric prefixes `sheriff.oauth` → `sheriff.token` (`JwtPropertyKeys.PREFIX`, `MetricIdentifier.PREFIX`, `application*.properties`, tests, docs).
* Requirement IDs `OAUTH-SHERIFF-N → VAL-N` (xref:../../Requirements.adoc[Requirements] anchors + all references).
* Docs, badges, README image `alt`, GitHub workflows, `.github/project.yml` Sonar key, `cuioss.github.io/OAuth-Sheriff` URLs.

*1d — Gate:* `./mvnw -Ppre-commit clean verify` then `./mvnw clean install` green.

== PR strategy (150-file review guideline)

The ~150-files-per-PR review limit is a *general guideline*, applied where sensible — not a hard gate for this change. Measured reality: this is one mechanical, tool-driven rename touching ~500 files (381 `.java` relocations + poms + ~425 `sheriff.oauth.` config-key files + docs).

* *PR 1 — build-safe textual rename (honors the guideline).* Documentation/prose, requirement IDs `OAUTH-SHERIFF-N → VAL-N`, badges, Sonar key, Pages URLs. No compile impact, genuinely reviewable; ~86 files (split into two if it exceeds 150).
* *PR 2 — atomic mechanical rename (guideline waived).* The OpenRewrite package/type/coordinate transform plus the `sheriff.oauth → sheriff.token` config/metric constants and their test expectations. This is *atomic* — packages, coordinates and config keys must flip together to keep `main` building. Slicing it into ≤150-file pieces would either break build-green-per-PR or leave coexisting `…oauth.*`/`…token.*` roots, adding risk with no review value.

*Why the waiver is the safe choice:* the guideline's intent is reviewable PRs. A 500-file *mechanical, idempotent, build-verified* diff is reviewed by inspecting the OpenRewrite recipe and the verification sweep — not by reading ~500 identical edits. Forcing a split here reduces safety rather than increasing it. (If the reviewer hard-caps file count, land PR 2 with review on the recipe + green CI, or split the package move by sub-package across stacked PRs accepting temporary mixed roots — only if mandated.)

== Maven Central relocation

Publish a final POM under each OLD coordinate whose `distributionManagement/relocation` points to the new coordinate. Resolvers auto-redirect to the new artifact and print a warning telling consumers to update.

[source,xml]
----
<project xmlns="http://maven.apache.org/POM/4.0.0">
  <modelVersion>4.0.0</modelVersion>
  <groupId>de.cuioss.sheriff.oauth</groupId>
  <artifactId>oauth-sheriff-core</artifactId>
  <version>0.9.0</version>
  <packaging>pom</packaging>
  <distributionManagement>
    <relocation>
      <groupId>de.cuioss.sheriff.token</groupId>
      <artifactId>token-sheriff-validation</artifactId>
      <message>Renamed to Token-Sheriff. See migration guide.</message>
    </relocation>
  </distributionManagement>
</project>
----

Publish a stub for *every* old artifact (`-core`, `-quarkus`, `-quarkus-deployment`, `-parent`, `bom`). Relocation redirects coordinates only — Java packages and config keys still require the consumer changes in xref:migration-nifi-extensions.adoc[the migration guide]. Reference: https://maven.apache.org/guides/mini/guide-relocation.html[Apache Maven — Guide to relocation].

== Risks & mitigations

* *Atomicity* — packages, config keys, and requirement IDs must flip together; a partial rename is worse than none. → One branch, OpenRewrite + scripted transforms, full build gate before merge.
* *Hidden string refs* — config/metric keys live in `.properties`, `.adoc`, and test literals, not only Java. → The scope counts above are the regression baseline; re-run expecting ~0.
* *External-service lag* — Sonar/Pages/OpenSSF update out of band. → Phase 3 checklist; verify each badge.
* *Consumer breakage* — `nifi-extensions` breaks when it bumps coordinates. → Relocation grace window + the migration guide.
