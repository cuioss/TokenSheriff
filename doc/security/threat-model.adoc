== Overview

_See Requirement xref:../Requirements.adoc#OAUTH-SHERIFF-8[OAUTH-SHERIFF-8: Security Requirements] and xref:../architecture.adoc[Architecture] and xref:security-reference.adoc[Security Reference]_

This document outlines the threat model for the JWT Token Validation library. The threat model follows OWASP recommendations and the STRIDE methodology. Each category lists threats, current mitigations with implementation references, coverage status, and open recommendations.

== System Context

The JWT Token Validation library:

* Processes JWT tokens from OAuth2/OpenID Connect providers
* Validates token signatures using JWKS (JSON Web Key Sets)
* Extracts and processes token claims
* Handles multiple token types (Access, ID, Refresh)
* Supports multiple issuers with per-issuer configuration
* Implements key management and rotation
* Supports DPoP sender-constrained tokens (RFC 9449)

== STRIDE Threat Analysis

=== Spoofing

_See Requirements xref:../Requirements.adoc#OAUTH-SHERIFF-8.1[OAUTH-SHERIFF-8.1] and xref:../Requirements.adoc#OAUTH-SHERIFF-8.4[OAUTH-SHERIFF-8.4]_

==== Threats and Mitigations

[cols="1,2,3,2", options="header"]
|===
|ID |Threat |Mitigation |Test Coverage

|S1
|Forged JWT tokens
|Cryptographic signature verification via link:../../oauth-sheriff-core/src/main/java/de/cuioss/sheriff/oauth/core/pipeline/validator/TokenSignatureValidator.java[TokenSignatureValidator]; asymmetric-only (HMAC/"none" rejected per RFC 8725)
|link:../../oauth-sheriff-core/src/test/java/de/cuioss/sheriff/oauth/core/pipeline/validator/TokenSignatureValidatorTest.java[TokenSignatureValidatorTest]

|S2
|JWKS endpoint impersonation
|TLS 1.2+ enforced for all JWKS communication via `SecureSSLContextProvider` (from cui-http-client dependency); HTTPS-only endpoints
|Integration tests verify HTTPS

|S3
|Expired/revoked token reuse
|Expiration (`exp`) and not-before (`nbf`) validation in link:../../oauth-sheriff-core/src/main/java/de/cuioss/sheriff/oauth/core/pipeline/validator/TokenClaimValidator.java[TokenClaimValidator]
|link:../../oauth-sheriff-core/src/test/java/de/cuioss/sheriff/oauth/core/pipeline/validator/TokenClaimValidatorTest.java[TokenClaimValidatorTest]

|S4
|Tokens from untrusted issuers
|Strict issuer allowlist in link:../../oauth-sheriff-core/src/main/java/de/cuioss/sheriff/oauth/core/IssuerConfig.java[IssuerConfig]; issuer resolved by link:../../oauth-sheriff-core/src/main/java/de/cuioss/sheriff/oauth/core/IssuerConfigResolver.java[IssuerConfigResolver]
|link:../../oauth-sheriff-core/src/test/java/de/cuioss/sheriff/oauth/core/IssuerConfigResolverTest.java[IssuerConfigResolverTest]

|S5
|Token substitution (client confusion)
|Audience (`aud`) + authorized party (`azp`) validation in link:../../oauth-sheriff-core/src/main/java/de/cuioss/sheriff/oauth/core/pipeline/validator/TokenClaimValidator.java[TokenClaimValidator]
|link:../../oauth-sheriff-core/src/test/java/de/cuioss/sheriff/oauth/core/ClientConfusionAttackTest.java[ClientConfusionAttackTest]
|===

==== Open Recommendations

* Token blacklisting for revoked tokens
* Rate limiting for token validation attempts
* JWK thumbprint validation for JWKS endpoints

|TLS 1.2+ enforced via `SecureSSLContextProvider` (cui-http-client); configurable minimum TLS version
|Integration tests verify HTTPS

* Constant-time comparison for security-critical comparisons

|ID |Threat |Mitigation |Test Coverage

|D2
|Large token processing
|Token size limit 8 KB enforced before any parsing in link:../../oauth-sheriff-core/src/main/java/de/cuioss/sheriff/oauth/core/pipeline/NonValidatingJwtParser.java[NonValidatingJwtParser]
|link:../../oauth-sheriff-core/src/test/java/de/cuioss/sheriff/oauth/core/pipeline/NonValidatingJwtParserTest.java[NonValidatingJwtParserTest]

|D3
|Complex token structures
|Payload size limit 8 KB, string size 4 KB, buffer 128 KB; safe JSON parsing
|link:../../oauth-sheriff-core/src/test/java/de/cuioss/sheriff/oauth/core/pipeline/NonValidatingJwtParserTest.java[NonValidatingJwtParserTest]

|D4
|Resource exhaustion (parallel requests)
|Thread-safe implementation; connection/read timeouts for HTTP operations
|Concurrency tests

|D5
|Cache poisoning
|JWKS from trusted HTTPS endpoints only; key rotation grace period with bounded retired key sets (default 3)
|link:../../oauth-sheriff-core/src/test/java/de/cuioss/sheriff/oauth/core/jwks/http/HttpJwksLoaderGracePeriodTest.java[HttpJwksLoaderGracePeriodTest]

|D6
|CPU exhaustion (crypto operations)
|Token validated only after size/structure checks pass; access token cache avoids re-validation
|Performance benchmarks
|===

==== Open Recommendations

* Request throttling per client
* Circuit breaker for JWKS endpoint failures

==== Open Recommendations

* Role hierarchy validation
* Mandatory audience validation for client applications
* Warning when mutable claims (e.g., `email`) used for identification

=== DPoP Threats (RFC 9449)

_See Requirement xref:../Requirements.adoc#OAUTH-SHERIFF-8.7[OAUTH-SHERIFF-8.7: DPoP Sender-Constrained Tokens]_

==== Threats and Mitigations

[cols="1,2,3,2", options="header"]
|===
|ID |Threat |Mitigation |Test Coverage

|DP1
|DPoP proof replay
|JTI-based replay protection with bounded cache (default 10,000 entries, 300s TTL) in link:../../oauth-sheriff-core/src/main/java/de/cuioss/sheriff/oauth/core/dpop/DpopReplayProtection.java[DpopReplayProtection]
|link:../../oauth-sheriff-core/src/test/java/de/cuioss/sheriff/oauth/core/dpop/DpopProofValidatorTest.java[DpopProofValidatorTest]

|DP2
|Token exfiltration (no sender binding)
|DPoP proof `jkt` (JWK Thumbprint) binding validates proof key matches access token's `cnf.jkt` claim
|link:../../oauth-sheriff-core/src/test/java/de/cuioss/sheriff/oauth/core/dpop/DpopPipelineIntegrationTest.java[DpopPipelineIntegrationTest]

|DP3
|Stale DPoP proof
|Proof `iat` validated against configurable max age (default 300s) in link:../../oauth-sheriff-core/src/main/java/de/cuioss/sheriff/oauth/core/dpop/DpopProofValidator.java[DpopProofValidator]
|link:../../oauth-sheriff-core/src/test/java/de/cuioss/sheriff/oauth/core/dpop/DpopProofValidatorTest.java[DpopProofValidatorTest]

|DP4
|DPoP algorithm confusion
|DPoP proof `alg` validated against allowed asymmetric algorithms; HMAC/"none" rejected
|link:../../oauth-sheriff-core/src/test/java/de/cuioss/sheriff/oauth/core/dpop/DpopProofValidatorTest.java[DpopProofValidatorTest]

|DP5
|HTTP method/URI mismatch
|DPoP proof `htm` and `htu` validation is not performed by the library; this is handled at the application/framework level (e.g., Quarkus integration)
|link:../../oauth-sheriff-core/src/test/java/de/cuioss/sheriff/oauth/core/dpop/DpopPipelineIntegrationTest.java[DpopPipelineIntegrationTest]
|===

==== Open Recommendations

* Server nonce support (`nonce` claim in DPoP proof)
* DPoP proof binding for refresh tokens
